# 04-案例：一周用 AI 做一个产品（完整记录）

> 读完本文你将了解：一个真实的「周一到周日」用 AI 做一个产品的完整过程、每一步具体做什么、用了什么 AI 工具、遇到了什么问题怎么解决、以及最后用户反馈如何
> 注：本文是一个完整的案例复盘，展示前 3 篇文章的方法论怎么在实战中落地。你不需要跟着做一遍，但要理解这个工作流为什么有效

---

## 案例背景：为什么选这个产品

为了写这一篇，我用一周时间做了一个真实的产品，并把完整过程记录了下来。

**产品定义**：一个叫 **RSS2Summary** 的 AI 工具——输入 RSS Feed URL，自动生成中文摘要，支持导出为 Markdown 邮件格式。

**为什么选这个产品**：
- **问题真实**：我自己就有这个需求——订阅了一堆技术 RSS，但每天根本看不过来，需要有人帮我先筛一遍
- **难度适中**：有技术挑战（RSS 解析 + LLM 摘要），但不会复杂到一周做不完
- **容易验证**：发到朋友圈和 RSS 爱好者社区就能拿到反馈

---

## 周一：问题定义与 Product Brief

### 上午：定义问题

先用「问题定义追问法」回答了几个关键问题：

```
我在解决的问题：技术内容消费者每天有大量 RSS 订阅内容要阅读，
但没有足够时间筛出真正有价值的文章，导致信息过载或干脆不看
```

3 个检验标准：
- ✅ 具体：每天打开 Feedly / FreshRSS 发现 200+ 条未读的人
- ✅ 高频：每天一次甚至更多——这是周期间歇的信息焦虑
- ✅ 够痛：信息过载是知识工作者的常态痛点，而且没有好的免费解决方案

### 下午：用户调研

我找了 5 个 RSS 使用者做了简短调研（每人 10 分钟）：

| 问题 | 回答模式 | 发现 |
|------|---------|------|
| 你每天有多少 RSS 未读？ | 100-500 条 | 信息量确实大，不是个例 |
| 你通常怎么处理未读？ | 要么全部标记已读，要么只看标题 | 大量内容没有真正被消化 |
| 你是不是已经放弃了 RSS？ | 3/5 人说「还在用但经常焦虑」 | 需求真实，且没有好解决方案 |
| 你试过什么工具解决？ | 试过 AI 摘要工具，但都是收费的 | 市场需求存在，付费意愿有待验证 |

### 结论

**核心假设**：用户需要一个免费、好用的 RSS 摘要工具，每天自动把订阅内容变成一键可读的摘要摘要。

**Product Brief 一句话定义**：为 RSS 订阅用户提供一个能自动抓取和摘要内容、以邮件/笔记形式分发的工具，解决信息过载问题。

## 周二：MVP 规划 + AI 生成原型

### 上午：MVP 需求清单

用 MoSCoW 法确定第一版的 Must 功能：

```
Must:
1. 用户可以输入 RSS Feed URL
2. 系统抓取最新 5-10 篇文章
3. 用 LLM 生成每篇文章的中文摘要（100-200 字）
4. 展示摘要列表，用户可以一键复制

Won't Have（第一版不做）:
- 用户注册系统 → 匿名使用
- 邮件定时推送 → 手动刷新
- 多语言支持 → 先只做中文摘要
- 历史记录 → 浏览器 localStorage
- 批量管理多个 Feed → 一次只处理一个
```

### 下午：用 AI 生成原型

**第一步：先不看 UI，用 AI 搭建核心逻辑**

```
Prompt：「帮我写一个 Python 脚本来抓取 RSS Feed。
使用 feedparser 库，解析后提取每篇文章的标题、链接、发布时间和正文。
输出为结构化的 JSON 格式。」
```

AI 生成了一个可用的 `rss_fetcher.py`——边写边修改，差不多用了 5 轮对话。

**第二步：生成前端原型**

```
Prompt：「用 HTML + Tailwind CSS 生成一个 RSS 摘要工具的页面原型。
布局：一个居中的输入框，用户贴 RSS URL 后点"获取摘要"，
下方展示文章列表，每篇显示标题、摘要内容（一段中文）和复制按钮。
使用 slate 配色，蓝色 accent。所有间距 4 的倍数。」
```

第一版 AI 生成的比较一般——我给了它几轮 refine 指令。关键改进包括：「增加 loading 状态」「把复制按钮移近文本」「让卡片 hover 时有微妙的阴影变化」。

### 当天成果

```
周一结束时的状态：
- 核心 Python 脚本可以抓取 RSS 内容
- 前端 HTML 原型可以在浏览器中打开
- 后端 API 还没接（暂时用 Mock 数据展示）
- 总耗时：约 6 小时
```

---

## 周三：核心功能开发（AI 主力）

这一天是全周代码量最大的一天，但实际写代码的是 AI，我做的是**设计接口、写 prompt、审查输出**。

### 上午：对接 AI 摘要

**需求**：把抓取到的文章正文传给 LLM，返回中文摘要。

```
Prompt：「写一个 Python 函数，调用 DeepSeek API 对一篇文章生成中文摘要（100-200 字）。
输入参数：title（字符串）、content（字符串）。
返回：summary（字符串）。
需要处理 API 错误和重试（最多 3 次）。」
```

AI 生成的代码需要注意几个细节：
- 初期 prompt 太短 → 摘要质量不好 → 加上了 system prompt：「你是一个专业的技术内容摘要者，请用平实的语言总结文章的核心观点和结论，100-200 字，不添加你的评论」
- 并发请求需要限流 → 让 AI 加了 `asyncio` + semaphore 控制

### 下午：前后端对接

**问题**：前端 HTML 原型和后端 Python 脚本是两个独立的东西。

**解决方案**：用 FastAPI 搭一个非常轻的后端：

```
Prompt：「写一个 FastAPI 应用，一个 API 端点 POST /summarize，
接收 JSON body：{"url": "https://..."}，
调用之前写的 rss_fetcher 和 summary_generator，
返回格式：{"articles": [{"title": "...", "link": "...", "summary": "..."}]}
添加 CORS 跨域支持。」
```

### 下午遇到的实际问题

| 问题 | 怎么解决的 |
|------|-----------|
| 某些 RSS Feed 格式不规范，feedparser 解析失败 | 让 AI 加了异常处理 + 回退到真接 HTTP 请求 |
| 文章正文有大量无关 HTML（导航、广告） | 让 AI 集成了 readability（html2text 库）提取正文 |
| DeepSeek API 偶尔超时 | AI 加了重试逻辑，并在前端展示 loading 动画 |

### 当天成果

```
周三结束时的状态：
- ✅ RSS 抓取（7 种常见 Feed 格式都能解析）
- ✅ AI 摘要（中文摘要质量通过自己测试）
- ✅ 前后端对接（前端可以输入 URL → 后端抓取 → 返回摘要）
- ❌ 尚未部署，只能在本地跑
- 总耗时：约 8 小时
```

---

## 周四：内部测试 + 迭代

### 上午：部署到线上

目标：今天让它跑在公网上，这样周五才能给用户测试。

**部署方案**：
- 前端：Vercel（部署静态 HTML，无需后端环境）
- 后端：Render（免费 tier，支持 Python FastAPI）

AI 帮忙写了 `vercel.json` 和后台的部署配置，不过部署踩了坑：
- Render 的免费实例会在空闲时休眠 → 第一次请求很慢 → 加了一个 wake-up 机制
- 前端 HLS 后端的 API URL 在前端是硬编码 → 改成通过环境变量配置

### 下午：内部测试 + bug 修复

我自己用它处理了 5 个 RSS Feed，发现：

| 发现的问题 | 严重程度 | 修复 |
|-----------|---------|------|
| 摘要里出现了未完成的句子 | 🔴 影响核心体验 | 增加了 token 限制和 prompt 约束 |
| 文章发布时间没有正确转换时区 | 🟡 体验问题 | 用 `pytz` 做了 UTC → 北京时间转换 |
| 复制按钮在手机上不像按钮 | 🟢 外观问题 | 留到第二版，第一版不影响使用 |
| 输入 +86 手机号的用户没地方填 | N/A | 这个不是目标用户群体，忽略 |

### 当天成果

```
周四结束时的状态：
- ✅ 产品已部署到公网（https://rss2summary.vercel.app）
- ✅ 核心流程通了：输入 URL → 等待 → 展示摘要
- ✅ 修复了 2 个核心 bug
- ⚠️ 第一次请求加载慢（因为服务器冷启动）
- 总耗时：约 5 小时
```

## 周五：给用户测试

### 上午：找测试用户

我做了 3 个动作：
1. 在 Twitter 和朋友圈发了一条：「我做了一个 RSS 摘要工具，免费，谁想试试」
2. 在几个 RSS 爱好者 Telegram 群发了链接
3. 直接私信了之前调研过的 5 个用户

### 下午：观察用户行为 + 收集反馈

关键发现（真实数据）：

| 指标 | 数据 | 解读 |
|------|------|------|
| 首日访问量 | 47 人 | 基础传播还行，但不算爆发 |
| 使用率 | 31/47（66%） | 点开的用户里 2/3 试用了——说明产品概念吸引人 |
| 重复使用 | 7/31（23%） | 一次以上的用户不多——核心假设部分成立但不够 |
| 留存 | 3/31（10%）次日活跃 | 留存率不理想，需要进一步分析原因 |

用户反馈整理：

| 反馈类型 | 具体内容 | 影响 |
|---------|---------|------|
| 正面 | 「摘要质量出乎意料地好」 | 核心功能通过了验证 |
| 正面 | 「复制到 Notion 很方便」 | 这个使用场景比预期更广 |
| 负面 | 「每次都要输入 URL，太麻烦了」 | 用户体验瓶颈——需要登录+保存 |
| 负面 | 「摘要更新频率不够」 | 用户想要定时推送 |
| 功能请求 | 「能不能顺便给文章打分？」 | 有价值但不是 MVP 要做的 |
| 负面 | 「服务器第一次加载太慢了」 | 冷启动确实影响体验 |

### 结论分析

```
核心假设验证结果：
✅ 用户确实有 RSS 信息过载的痛点（确认）
✅ AI 摘要质量经过用户认可（确认）
⚠️ 用户的「使用频率」低于预期（需要迭代）
⚠️ 用户的「保存和回访」需求比「摘要生成」更强（方向需要调整）

决策方向：Iterate（非 Pivot）
核心功能成立，但用户交互方式需要改进
```

### 当天成果

```
周五结束时的状态：
- ✅ 获得了 47 个实际用户的第一手反馈
- ✅ 确认了核心功能（AI 摘要）的质量通过用户验证
- ✅ 发现了 3 个关键的改进方向
- 总耗时：约 4 小时
```

## 周六：基于反馈迭代

### 上午：开发 V1.1

基于周五的反馈，做了以下改进：

1. **新增保存功能**：用 localStorage 保存用户已经生成过的 Feed URL 和摘要历史
2. **增加 Feed 列表功能**：用户可以保存多个 Feed，一键切换
3. **优化冷启动**：加了健康检查页面，用固定间隔 ping 服务器保持活跃

AI 在这天的开发里效率极高，因为已经有了清晰的反馈和需求：

```
Prompt：「现有代码（粘贴）。用户反馈想做这样一个功能：
用户输入的 RSS URL 可以保存在 localStorage 里，下次打开还能看到。
需要在前端增加"保存"按钮和"最近 Feed"列表。后端不需要改。」
```

AI 一次生成就基本可用，只做了一轮微调。

### 下午：再次部署 + 预告

部署了 V1.1，然后在群里发了更新说明。同时把周一的 Product Brief 改成了：

- **更新后的核心假设**：用户需要一个可以保存管理多个 RSS Feed、每天自动更新的摘要工具
- **新增 Won't Have**：手机 App、邮件发送、多语言

### 当天成果

```
周六结束时的状态：
- ✅ V1.1 迭代上线：新增保存 + Feed 列表
- ✅ 冷启动问题有改善
- ✅ 用户已经有了回访的价值（保存了 Feed 会想回来看）
- 总耗时：约 4 小时
```

## 周日：复盘 + 写文档 + 思考下一步

### 上午：写项目复盘

一周做完的 MVP 不会完美，但已经很值得复盘了：

**数据汇总（一周）**：

| 指标 | 数字 |
|------|------|
| 总开发时间 | ~27 小时 |
| 总代码行数 | ~1,200 行（前端 500 + 后端 700） |
| 其中 AI 生成的代码 | ~1,100 行（92%） |
| 用户访问量 | 47 人 |
| 试用人次 | 31 人 |
| 生产环境中发现的 bug | 3 个 |
| MVP 后 to-do 清单项 | 7 条 |

**最大的收获**：

1. **调研节省了试错成本**——周一花了半天做调研，直接决定了产品方向的对错判断标准
2. **Won't Have 清单救了命**——如果没有明确不做注册系统，开发时间至少多 50%
3. **AI 让「一周一个」成为可能**——92% 的代码由 AI 生成，我真正花时间的是：定义问题、设计接口、审查代码质量和分析用户反馈
4. **用户反馈是最值钱的东西**——周五得到的 3 条核心反馈，比我自己闷头做三周改进都值钱

### 下午：思考下一步

```
产品的下一步不是继续大规模开发，而是做第二个 MVP 来验证新的假设：

新假设：用户最大的需求不是「摘要」，而是「每天有人帮我筛选和推送」

验证方案：做一个 Telegram Bot 版本，每天定时推送摘要到用户的 Telegram
不需要前端，不需要注册——用 Telegram ID 做用户标识
如果用户在 Telegram bot 的留存率 > 50%，说明推送模式才是对的
```

## 全文总结：一周开发的工作流

```
周一  →  定义问题 + 用户调研 + Product Brief
周二  →  MVP 规划 + AI 生成原型（Mock 数据）
周三  →  核心功能开发（AI 主力，你做接口和审查）
周四  →  内部测试 + bug 修复 + 部署
周五  →  给用户测试 + 收集反馈
周六  →  基于反馈做 V1.1 迭代
周日  →  复盘 + 文档 + 决定下一步
```

### 这个工作流能成功的关键

| 原则 | 为什么重要 |
|------|-----------|
| **时间盒** | 一周是个硬约束，逼你做最核心的事。不要等到完美再上 |
| **先验证再完善** | MVP 的目标是验证假设不是交付完整产品。周三的核心功能够了就上 |
| **AI 是加速器，不是决策者** | AI 帮我实现了 92% 的代码，但我仍然花了大量时间做「不该做的事」决策 |
| **用户反馈是一手资料** | 47 人试用得到的 3 条核心反馈，比 100 页市场报告都值钱 |
| **周末复盘是闭环** | 一周结束不是结束，是下一个 Build-Measure-Learn 循环的开始 |

### 你现在应该做什么？

1. 找一个你最近想做（或者以前想过但没做）的产品想法
2. 本周一就做 Product Brief，不要等到「准备充分」
3. 按上面的节奏做一周，每天设定明确的目标
4. 周五一定要发给真用户测试——不要怕丢脸，越早上线越早学习
5. 周日写复盘，决定下一步是 iterate、pivot 还是 kill

**要点总结**
- 一周做一个产品的节奏是可行的，前提是：**严格定义问题、砍掉所有非核心功能、AI 做主力开发**
- 周一周二花在规划调研上，周三周四集中开发，周五给用户测试，周六迭代，周日复盘
- **92% 的代码由 AI 生成**，但关键决策（做什么、不做什么、判断用户反馈）全由人做
- MVP 的目标不是完美，是**用最小的成本拿到最有价值的学习**
- 一周结束不是终点，是**下一个 Build-Measure-Learn 循环的起点**
- 越早上线，越早学习。不要等完美——等完美的代价是从来没有开始
