# 02-MVP 与快速验证：最小成本试错的工作哲学

> 读完本文你将了解：MVP 到底是什么（和你以为的可能不一样）、怎么把需求砍到只剩最核心的部分、如何用 AI 大幅降低 MVP 的开发成本、以及验证假设后怎么决定下一步
> 注：本文的核心不是「怎么做 MVP 开发」，而是**怎么用 MVP 的思维方式来降低试错成本**——特别是结合 AI 能力之后

---

## 你可能遇到过的问题

你终于有了一个清晰的产品想法，开始动手了，但很快又陷入了困境：

- **功能越做越大，越做越慢**——原本说好两周做完的东西，做着做着就膨胀成两个月了
- **舍不得砍功能**——每一个功能都觉得有必要，砍掉任何一个这产品就不完整了
- **所有时间都花在完美主义上**——登录页面的按钮对齐调了三个小时，核心功能还一句代码没写
- **做完了发现没人用**——吭哧吭哧做了几个月上线的产品，用户量不如随便发的一条 Twitter
- **AI 写代码效率太高，反而不知道优先写什么**——反正 AI 写得快，一口气把所有功能都做了，结果发现做了很多没人用的功能

这些问题的核心原因只有一个：**你搞错了 MVP 的真正含义**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| MVP 就是做一个功能残缺的半成品 | MVP 是**功能最少但价值完整的**版本——去掉的是功能数量，不是用户体验质量 |
| MVP 就是快速开发、快速上线 | MVP 的核心是「快速验证」，不是「快速发布」——发布速度是手段，获得反馈才是目的 |
| MVP 只适合创业公司 | 任何新项目都可以用 MVP 思维：新功能、新模块、新工具，先小范围跑通再扩大 |
| MVP 做好了就可以一直用下去 | MVP 是**一次性学习工具**，验证完假设就该迭代或 pivot，不是永远维持原样 |
| 有 AI 了，可以直接从完整版做起 | AI 让开发变快，但**验证的价值没有变**——做错了方向只是比以前更快浪费资源 |

---

## 常见问题快速解决

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 功能清单越写越长 | 用 MoSCoW 法给每个功能标 Must / Should / Could / Wont，第一版只做 Must |
| 不确定 MVP 该做什么 | 回到上一篇文章的 Product Brief，只做**验证核心假设**需要的功能，不是「完整产品」需要的功能 |
| AI 帮你写了太多功能代码 | 禁用 AI 生成新功能，只让它完善当前 MVP 的核心功能 |
| 做完美主义犹豫不前 | 设一个硬性截止时间——「这周六必须上线一个可用版本」，时间到了就发 |
| 不知道 MVP 做完之后怎么做 | MVP 做完只是拿到反馈数据；反馈决定了是 iterate、、还是 pivot、还是 kill |

## MVP 的核心：验证假设，不是交付功能

Eric Ries 在 *The Lean Startup* 里定义了 MVP 的真正目的：

> 一个 MVP 的版本可以帮助你以**最小的努力**、**最短的反馈循环**来验证你的核心假设。

理解这句话的关键是分清两个概念：

| | 传统开发思维 | MVP 思维 |
|-------------|------------|---------|
| 目标 | 交付一个功能完整的产品 | 学习一个核心假设是否成立 |
| 成功标准 | 功能全部上线 | 获得了足够的数据来做下一步决策 |
| 时间观 | 按里程碑推进 | 按「学习周期」推进（Build → Measure → Learn） |
| 对待失败 | 失败了就是项目失败 | 失败是**有效的学习**——证明了假设不成立 |

### Build-Measure-Learn 循环

```
┌───────┐   ┌─────────┐   ┌────────┐
│ Build │ → │ Measure │ → │ Learn  │ ──→ 下一个循环
└───────┘   └─────────┘   └────────┘
    ↑                                    │
    └────────────────────────────────────┘
```

每一次循环的目标都是「学到一个新信息，然后用这个信息决定下一步」。MVP 是**加速这个循环的工具**，不是一次性的交付物。

## 怎么砍需求：MVP 需求优先级的 4 种方法

砍需求是最难的事。每个功能都有它存在的「道理」，但不是每个功能都值得在第一版做。

### 1. MoSCoW 优先级法

```
Must have（必须有）    →  没有这个功能，核心假设就无法验证
Should have（应该有）  →  很重要，但可以没有
Could have（可以有）   →  有更好，没有不致命
Won't have（这版不做） →  明确决定放弃
```

**关键原则**：第一版只做 Must。Should 以下的全部砍掉，放到迭代清单里。

### 2. 价值-成本矩阵

| | 低实现成本 | 高实现成本 |
|---------|-----------|-----------|
| **高验证价值** | ✅ **第一版必须做** | ⚠️ 能做就做，不能就简化 |
| **低验证价值** | ❌ 浪费，不做 | ❌❌ 坚决不做 |

**判断标准**：不是「这个功能有没有用」，而是「**不做这个功能，核心假设还能不能验证**」。

### 3. 用户旅程倒推法

画出用户从第一次打开产品到完成核心任务的旅程，只保留那段旅程里**必须出现**的功能。其他全部砍掉。

```
用户旅程示例（AI 播客摘要工具）：
1. 打开网站 → 看到输入框
2. 粘贴播客 URL → 点击生成
3. 等待几秒 → 看到摘要
4. 阅读、复制 → 离开

第一版只需要这些功能：
- ✅ 输入框
- ✅ 生成按钮
- ✅ 摘要展示区
- ✅ 复制功能

不需要的功能：
- ❌ 用户注册登录
- ❌ 历史记录
- ❌ 分享功能
- ❌ 收藏功能
- ❌ 多语言界面
```

### 4. 「这版本不做」清单

比「做什么」更重要的是「明确不做什么」。**写一个 Won't Have 清单，贴在项目根目录**，每次想加功能先看这个清单。

```
# 第一版不做的功能
- 用户注册/登录（不收集个人信息，零门槛试用）
- 历史记录（本地存 Cookie 即可）
- 移动端适配（先只优化桌面端）
- 付费功能（全部免费，数据就是你的报酬）
- 批量处理（一次只处理一篇）
```

写完这个清单后，如果有人建议加功能，先问：「这个功能不在 Won't Have 清单上，但它是 Must 吗？没有它核心假设验证不了吗？」

## AI 怎么加速 MVP 开发（具体案例）

AI 是 MVP 开发者的最大红利。以下是几个具体的应用场景：

### 场景 1：快速原型 — 从 Product Brief 到可运行的原型

```
Prompt：「根据这个 Product Brief，帮我生成一个单页面 HTML 原型的代码。
只需要一个输入框、一个生成按钮、一个结果展示区。用 Tailwind CSS 写样式，
部署到 Netlify 可以直接使用。」
```

不需要写任何前端代码，10 分钟拿到**可以给用户试用的**原型。虽然代码质量不完美，但它足够验证核心假设了。

### 场景 2：不做注册系统

MVP 阶段**不应该有用户注册系统**。注册是最大的用户流失点——用户还没验证你的产品有没有价值，就要先给一堆个人信息。

替代方案：
- 匿名使用 + UUID 标识会话
- 本地 localStorage 存储用户上次的输入
- Landing page 上放一个「立刻试用」按钮，直接进入主功能

AI 提示词：「生成一个匿名使用的 AI 工具，不需要用户注册，用 localStorage 记录用户的使用记录」

### 场景 3：低保真界面先验证流程

不用等到 UI 完美再上线。用 HTML + Tailwind + 一些基本样式就行。重点验证的是**流程走得通、用户能理解、回答有质量**。

**AI 提示词**：「用 HTML + Tailwind CSS 做一个播客摘要工具的页面原型，风格简洁，只保留输入框、生成按钮、结果展示三个区域。不要任何装饰性元素。」

### 场景 4：Mock 数据验证核心体验

如果核心功能依赖复杂的后端逻辑（比如 RAG pipeline），MVP 阶段可以先 Mock。Mock 的意思是：暂时不接入真实的大模型 API，用预设的假数据模拟输出，让你先验证**用户体验**和**交互流程**。

```
AI 提示词：「帮我写一个播客摘要工具的 Mock 版本。用户输入 URL 后，
前端模拟 2 秒延迟，然后展示一段预设的摘要文本。不需要真实 API 调用。」
```

这样你可以在后端还没完成时，先给用户看前端体验，收集对摘要格式、展示方式、交互流程的反馈。

## MVP 做完之后怎么走：3 种可能

MVP 不是终点，是起点。拿到用户反馈后，你有 3 种选择：

### 1. Iterate：假设成立，继续打磨

如果用户用你的 MVP，而且愿意继续用：
- 数据显示用户活跃度超过预期
- 用户主动提供了反馈和 feature request
- 你验证了「用户真的需要这个功能」

**下一步**：基于反馈做下一轮迭代，补上 Should 和 Could 功能。

### 2. Pivot：假设部分成立，但方向不对

如果用户有一些使用行为，但核心假设没有完全验证：
- 用户用了，但用在了你没有预想到的场景
- 核心功能没人用，但某个边角功能反而有人喜欢
- 用户说不错，但就是不再打开第二次

**下一步**：重新定义问题，调整方向，做新的 MVP 重新验证。

### 3. Kill：假设不成立，果断放弃

如果用户根本不使用、或者用了一次就丢掉：
- MVP 上线一周日活 < 5
- 用户反馈「没解决我的问题」
- 用户尝试了，但没有留下任何数据

**下一步**：果断放弃，把经验和教训写下来，开始下一个项目。MVP 的使命就是「用最小的成本告诉你这个想法行不行」。**放弃不是失败，是有效的学习**。

## 一个 AI 加速 MVP 的工作流

```
周一    周二    周三    周四    周五    周六
  ↓       ↓       ↓       ↓       ↓       ↓
Product   AI 生   核心功   内部测   给用户   收集反
 Brief → 成原型 → 能开发 → 试迭代 → 测试用 → 馈决策
```

1. **周一**：写清楚一页纸 Product Brief，明确核心假设
2. **周二**：用 AI 生成 UI 原型和 Mock 数据，先验证流程
3. **周三**：用 AI 完成核心功能的代码，只做 Must
4. **周四**：内部跑通流程，修复明显问题和 bug
5. **周五**：发给 10-20 个目标用户试用，不要求注册
6. **周六**：收集反馈数据，决定 iterate / pivot / kill

这个节奏的核心是：**在 5 天内完成一次完整的 Build-Measure-Learn 循环**。AI 把以前需要 1 个月的开发周期压缩到了一周。

---

## 你现在应该做什么？

1. 拿出你上一篇文章写的 Product Brief
2. 用 MoSCoW 法标出第一版真正的 Must 功能（不超过 3 个）
3. 写一份 Won't Have 清单，贴在项目根目录
4. 用 AI 生成一个可运行的原型——今天就做，不用等到「准备充分」
5. 给自己设一个截止日期（建议 3-5 天内），到了就上线给用户用

**要点总结**
- MVP 的核心目的不是交付功能，是**验证核心假设**，用最小的成本获得最大的学习
- Build-Measure-Learn 循环是 MVP 的底层方法论——每个 MVP 都是一个加速的学习工具
- 砍需求的 4 种方法：**MoSCoW、价值-成本矩阵、用户旅程倒推、Won't Have 清单**
- AI 在 MVP 阶段特别有用：快速原型、跳过注册系统、Mock 数据、快速迭代
- MVP 后的 3 种选择：**Iterate（继续）、Pivot（转向）、Kill（放弃）**，每一种都是有效的学习
- 用 AI 可以把传统一个月的 MVP 周期压缩到一周以内——降低试错成本才是 MVP 的最大价值
