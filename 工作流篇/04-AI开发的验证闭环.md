# 04-AI开发的验证闭环

> 读完本文你将了解：为什么 AI 开发需要「验证闭环」而不是「一次生成」，怎么在每步实施前定义验证标准，用 enzed/vibe-coding 的实践模式做 Step-by-Step 验证开发，以及测试驱动开发（TDD）在 AI 时代的变体

---

## 你可能遇到过的问题

- **你让 AI 写了一个完整模块，跑起来一堆 Bug，改了一个另一个又炸了**
- **AI 改代码时破坏了原来能用的功能，但你到上线前才发现**
- **你不知道 AI 生成的代码到底对不对，只能赌一把**
- **测试是上线前才补的，结果发现核心逻辑有根本问题，要重写**
- **你告诉 AI「帮我加个功能」，它改完看起来没问题，但你不敢点「确认」**

这些问题有一个共同根源：**你把 AI 生成的结果当成了终点，而不是验证的起点**。AI 开发的工作流不应该是一步到位式的「写一大段、再修 Bug」，而应该是小步迭代式的「写一点、测一点、验证一点」。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 先让 AI 写完所有功能，最后再测试 | AI 代码的错误是分散的，等写完了再测，改起来成本极高 |
| 既然 AI 生成的代码，让它自己测就行 | AI 不擅长发现自己的逻辑盲区，需要你定义验证标准 |
| 测试是 QA 的事，不关开发的事 | AI 时代测试是你的安全网——没有测试你不敢让 AI 改代码 |
| 验证就是看看输出对不对 | 验证包括测试、类型检查、Lint、安全扫描、性能基准 |
| 项目做完再补测试 | 验证应该是每一步都有，不是最后才做的 |

---

## 常见问题快速解决

| 你遇到的问题 | 快速操作 |
|-------------|---------|
| AI 改代码破坏原有功能 | 为核心功能写 Regression Test，每次改完就跑一遍 |
| 不知道 AI 的代码对不对 | 先让 AI 写测试，再让 AI 写实现代码，用测试验证 |
| 验证太慢不想做 | 从最小验证开始：先加一个类型检查，再补一个核心测试 |
| 不知道该验证什么 | 用「输入示例 → 预期输出」定义验收标准，写进测试 |

## 什么是验证闭环

验证闭环的核心思想很简单：**每完成一步，跑测试验证，验证通过才进入下一步**。这在传统软件开发里叫 TDD（Test-Driven Development），但在 AI 开发时代，它的形式变了：

传统 TDD：写测试 → 写实现 → 验证通过 → 重构
AI 时代 TDD：定义验收标准 → AI 生成 → 自动验证 → 不通过则迭代 → 通过后继续

区别在于，传统 TDD 的「写实现」是你自己写，AI 时代是 AI 生成。但验证这一步依然是人类的职责——你来定义「什么算通过」。

## enzed/vibe-coding 模式

一位开发者[enzed](https://github.com/enzzed/vibe-coding)实践了一种高效的 AI 开发工作流，核心原则是：**分解成最小可验证步骤，每步验证通过后再继续**。

### 核心流程

```
需求分析 → 分解成小任务
         ↓
    ┌─────────────────────────┐
    │ Step N: 选择一个最小任务  │
    │   ↓                     │
    │ AI 生成代码              │
    │   ↓                     │
    │ 自动验证：               │
    │  • Lint 检查             │
    │  • 类型检查              │
    │  • 测试通过              │
    │  • 安全扫描              │
    │   ↓                     │
    │ ┌─ 不通过 → 返回修复 ──┐ │
    │ │                      │ │
    │ └── 通过 → 提交代码 ───┘ │
    └─────────────────────────┘
         ↓
    所有步骤完成 → 集成验证
```

### 关键原则

1. **步骤足够小**：每个 Step 应该在 5-10 分钟能完成验证。如果一次验证超过 10 分钟，说明步骤分解不够细
2. **验证先行**：在 AI 写代码之前，先定义验收条件。让 AI 先写测试，再写实现
3. **不通过不前进**：如果当前步骤验证不通过，不进入下一步。这和传统的「先堆积代码、最后再查 Bug」正相反
4. **提交即检查点**：每个验证通过的步骤都要提交，这样随时可以回退到最近的可工作状态

## 验证闭环的四层验证

一个完整的验证闭环应该包含以下四层，从上到下依次执行：

### 第一层：代码质量验证（秒级）

| 验证项 | 工具 | 检查什么 |
|--------|------|---------|
| 语法检查 | ruff / flake8 | 语法错误、未使用的导入 |
| 类型检查 | mypy / pyright | 类型注解是否正确 |
| 格式化 | ruff format / black | 代码风格是否统一 |
| 静态分析 | bandit | 常见安全漏洞 |

在 Step 开始时配置一次，后续每次 AI 生成代码后自动执行：

```bash
# 一键执行
ruff check . && mypy . && ruff format . --check
```

### 第二层：单元测试验证（秒级到分钟级）

这是验证闭环的核心。AI 写代码时最擅长「逻辑对但细节错」，单元测试是捕获细节错误的最高效手段。

**实用工作流**：

1. 你定义验收条件：「输入 X 时，函数应该返回 Y」
2. AI 先生成测试（基于你的验收条件）
3. 你 Review 测试是否正确
4. AI 再生成实现代码
5. 运行测试验证

```python
# 验收条件：process_order 函数
# - 正常订单返回 status: "processed"
# - 金额为 0 的订单返回 status: "rejected"
# - 用户黑名单返回 status: "blocked"

# Step 1: AI 写测试
def test_process_order_normal():
    assert process_order({"user_id": 1, "amount": 100})["status"] == "processed"

def test_process_order_zero_amount():
    assert process_order({"user_id": 1, "amount": 0})["status"] == "rejected"

def test_process_order_blacklisted():
    assert process_order({"user_id": 999, "amount": 100})["status"] == "blocked"

# Step 2: AI 写实现
def process_order(order):
    if is_blacklisted(order["user_id"]):
        return {"status": "blocked"}
    if order["amount"] <= 0:
        return {"status": "rejected"}
    return {"status": "processed"}

# Step 3: 跑测试验证
# pytest test_order.py  # 通过→进入下一项
```

### 第三层：集成验证（分钟级）

验证当前步骤的代码和已有代码是否能协同工作：

```bash
# 运行所有测试
pytest

# 检查模块间的接口是否匹配
# 比如新加的 API 端点和已有的路由系统是否兼容
```

### 第四层：行为验证（可选的，分钟级到小时级）

对于复杂的多步骤功能，可以用 E2E 测试或用 AI 做行为验证：

```
Prompt: "检查应用是否能完成以下用户流程：
1. 用户注册
2. 创建第一个项目
3. 邀请协作者
4. 提交变更

如果在某步出错，指出是哪步、错误是什么。"
```

## 验证闭环代码示例

一个完整的验证脚本示例，你可以放到项目根目录。每次 AI 生成代码后运行：

```python
"""run_verification.py — AI 开发验证脚本"""

import subprocess
import sys

def run_command(cmd, description):
    print(f"\n{'='*50}")
    print(f"🔍 {description}")
    print(f"{'='*50}")
    result = subprocess.run(cmd, shell=True)
    if result.returncode != 0:
        print(f"❌ {description} 未通过")
        sys.exit(1)
    print(f"✅ {description} 通过")
    return result

def main():
    # Level 1: 代码质量
    run_command("ruff check .", "Lint 检查")
    run_command("mypy .", "类型检查")
    run_command("ruff format . --check", "格式检查")

    # Level 2: 单元测试
    run_command("pytest --cov=./ --cov-report=term", "单元测试")

    print("\n🎉 所有验证通过！")

if __name__ == "__main__":
    main()
```

每次 AI 生成代码后，直接把这段脚本跑一遍。不通过就让 AI 修复，通过了再继续。

## 验证闭环的三大场景

### 场景一：从零构建新功能

```
Step 1: 定义 API 接口（先写接口文档）
  → 验证：接口文档 Review
Step 2: AI 生成接口实现
  → 验证：类型检查 + 单元测试
Step 3: AI 生成数据库操作
  → 验证：单元测试（Mock DB）
Step 4: 集成
  → 验证：集成测试
```

### 场景二：AI 修改已有代码

```
Step 1: 运行全部现有测试（确保当前是绿色状态）
  → 验证：全部测试通过
Step 2: 描述修改目标，让 AI 生成修改方案
  → 验证：方案 Review
Step 3: AI 执行修改
  → 验证：全部测试通过（Regression） + 新增测试通过
```

### 场景三：AI 修复 Bug

```
Step 1: 写一个能复现 Bug 的测试（先让这个测试失败）
  → 验证：新测试失败
Step 2: 让 AI 修复代码
  → 验证：新测试通过 + 全部旧测试通过
```

这种「先写失败测试，再修复」的方式，确保你真正理解了 Bug，而且修复是稳定的。

## 验证闭环的工具链推荐

| 类别 | 工具 | 用途 |
|------|------|------|
| Lint | ruff | Python 极速 Lint |
| 类型检查 | mypy / pyright | 静态类型验证 |
| 测试框架 | pytest | 单元/集成测试 |
| 测试覆盖率 | pytest-cov | 覆盖率报告 |
| 安全扫描 | bandit | 安全漏洞检测 |
| CI/CD | GitHub Actions | 自动验证流水线 |
| Pre-commit | pre-commit hooks | Commit 前自动验证 |

### Pre-commit 配置

在 `.pre-commit-config.yaml` 设置，确保每次 Commit 前自动验证：

```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.9.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.13.0
    hooks:
      - id: mypy
```

配置后，每次 `git commit` 时会自动跑这些检查，不通过无法提交——这就是验证闭环最直接的落地方式。

## 你现在应该做什么？

1. 选一个你正在做的 AI 项目，把任务分解成 5-10 个最小步骤
2. 为当前步骤写一个测试（验收条件），再去让 AI 写实现
3. 在项目里配置 Ruff + Mypy + Pytest，写一个 `run_verification.py`
4. 配置 Pre-commit Hooks，让每次 Commit 前自动验证

---

**要点总结**
- 验证闭环的核心思想是「每完成一步，跑测试验证，通过才进入下一步」
- AI 时代 TDD 的变体：定义验收标准 → AI 生成 → 自动验证 → 不通过则迭代
- enzed/vibe-coding 模式要求将任务分解成 5-10 分钟能验证的步骤，每步提交即检查点
- 四层验证：代码质量（秒级）→ 单元测试（秒级到分钟级）→ 集成验证 → 行为验证
- Plubot 模式：先写失败测试（复现 Bug/定义新功能），再让 AI 修复实现
- Pre-commit Hooks 是验证闭环最直接的落地方式，不通过无法提交
