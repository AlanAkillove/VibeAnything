# 05-用AI做项目的完整案例

> 读完本文你将了解：用一个真实的 TODO API 服务项目，展示从需求分析、设计文档、AI 实现、测试验证到迭代优化的完整过程，以及每一步 AI 是怎么用的、验证是怎么做的

---

## 项目概述

我们做一个「团队 TODO API」服务。这个项目不大不小——足够展示完整的开发流程，又不至于篇幅太长。

**核心功能**：
1. 用户注册和登录（JWT 鉴权）
2. 创建/查看/更新/删除 TODO 项目（CRUD）
3. 每个项目支持多个任务
4. 任务状态管理（待办/进行中/已完成）
5. 基本的权限控制（只能操作自己的项目）

**技术选型**：FastAPI + SQLite + Pydantic + pytest

这个案例展示了 AI 开发的完整工作流，每一步都用 AI 辅助，每一步都有验证闭环。

---

## 第一阶段：需求分析与设计文档

### 先写设计文档，再写代码

很多人用 AI 开发的习惯是直接说「帮我写一个 TODO API」，AI 生成了就用——但这样做出来的东西往往是拼凑的，缺乏整体设计。

正确做法：**先让 AI 帮你写设计文档，Review 确认后再开始写代码**。

### 给 AI 的 Prompt

```
我想做一个团队 TODO API 服务，功能包括：
1. 用户注册登录（JWT）
2. TODO 项目 CRUD
3. 任务管理（属于项目，有状态）
4. 权限控制（只能操作自己的项目和任务）

请帮我写一个设计文档，包括：
- API 端点设计（请求/响应格式）
- 数据库表结构
- 数据流图
- 目录结构
```

### AI 生成的设计文档（摘要）

**API 端点设计**：

| 方法 | 路径 | 功能 | 鉴权 |
|------|------|------|------|
| POST | /api/register | 用户注册 | 否 |
| POST | /api/login | 用户登录 | 否 |
| GET | /api/projects | 获取我的项目列表 | 是 |
| POST | /api/projects | 创建项目 | 是 |
| GET | /api/projects/{id} | 查看项目详情 | 是 |
| PUT | /api/projects/{id} | 更新项目 | 是 |
| DELETE | /api/projects/{id} | 删除项目 | 是 |
| GET | /api/projects/{id}/tasks | 查看项目任务列表 | 是 |
| POST | /api/projects/{id}/tasks | 创建任务 | 是 |
| PUT | /api/tasks/{id} | 更新任务状态 | 是 |
| DELETE | /api/tasks/{id} | 删除任务 | 是 |

**数据库表结构**：

```
users: id, username, email, password_hash, created_at
projects: id, user_id(FK), name, description, created_at, updated_at
tasks: id, project_id(FK), title, description, status(enum), created_at, updated_at
```

**目录结构**：

```
todo-api/
├── app/
│   ├── __init__.py
│   ├── main.py          # 应用入口
│   ├── models.py        # SQLAlchemy 模型
│   ├── schemas.py       # Pydantic Schema
│   ├── auth.py          # JWT 鉴权
│   ├── crud.py          # 数据库操作
│   └── routers/
│       ├── __init__.py
│       ├── auth.py      # 认证路由
│       ├── projects.py  # 项目路由
│       └── tasks.py     # 任务路由
├── tests/
│   ├── __init__.py
│   ├── conftest.py      # 测试配置
│   └── test_api.py      # API 测试
├── requirements.txt
└── run_verification.py
```

### 设计 Review

设计文档写完后，Review 以下要点：
- 端点是否覆盖所有需求？
- 数据结构是否合理？
- 鉴权策略是否正确？

Review 通过后，进入实现阶段。

```python

def test_auth():
    from app.auth import create_access_token, verify_token, hash_password, verify_password
    
    # 密码哈希测试
    hashed = hash_password("test123")
    assert verify_password("test123", hashed)
    assert not verify_password("wrong", hashed)
    
    # Token 测试
    token = create_access_token({"sub": "1"})
    payload = verify_token(token)
    assert payload["sub"] == "1"
    
    print("Auth OK ✅")

test_auth()
```

### Step 9：完善错误处理

1. 部分端点没有 500 错误的 try-catch

## 完整工作流回顾

```
需求分析 ───► 设计文档 ───► Review ───► 分步骤实现
                               │              │
                               │              ▼
                               │         每步验证
                               │              │
                               │              ▼
                               │       ┌─ 不通过 ─► AI 修复
                               │       │
                               │       └── 通过 ──► 提交 + 下一步
                               │              │
                               │              ▼
                               └─────────► 集成测试
                                              │
                                              ▼
                                        全部通过？
                                              │
                                    ┌──── 是 ─┴─ 否 ────┐
                                    │                    │
                                    ▼                    ▼
                                部署上线              修复迭代
```

**时间统计**：
- 设计阶段：15 分钟（AI 写设计文档 + 人工 Review）
- 实现阶段：45 分钟（10 个 Step，每个约 4-5 分钟）
- 测试阶段：20 分钟（写测试 + 修复发现的 2 个 Bug）
- 迭代优化：20 分钟（加分页 + 搜索）
- 总计：约 1 小时 40 分钟

如果没有验证闭环，直接让 AI 一次性生成全部代码：
- AI 生成：5 分钟
- 跑起来发现问题：30 分钟
- 逐 Bug 修复：1 小时起步
- 总时间：约 2 小时，而且你不知道还有多少隐藏 Bug

验证闭环多花了验证的时间，但节省了数倍的 Debug 时间，而且你对自己的代码有信心。

## 你现在应该做什么？

1. 选一个你想用 AI 做的项目（不必是 TODO API，可以是任何功能）
2. 先让 AI 写设计文档，Review 后开始实现
3. 把项目分解成 5-10 个最小步骤，每步验证通过再继续
4. 参照本文的 `run_verification.py`，在项目里配置验证脚本
5. 记录你的时间和 Bug 数，看看验证闭环是不是真的帮你省了时间

---

**要点总结**

1. 用 AI 做项目的正确姿势：需求整理 → AI 写设计文档 → 你 Review 确认 → 分步骤实现（每步验证） → 集成测试 → 迭代优化
2. 先写设计文档再写代码——设计阶段花 15 分钟 Review 能省下实现阶段数小时的返工
3. 每个功能拆分成 5-10 分钟可验证的最小步骤——每步 Lint + 类型检查 + 测试验证，不通过不前进
4. 测试发现的 Bug 让 AI 修复后重新验证——确保修复不引入新问题
5. 本文完整案例总用时约 1 小时 40 分钟，比一次性生成再修 Bug 的模式更高效且更可靠
6. Pre-commit 配置和验证脚本（run_verification.py）是项目基础设施——一次配置，长期受益
