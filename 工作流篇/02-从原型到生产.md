# 02-从原型到生产

> 读完本文你将了解：AI 做的 Prototype 和可上线的代码之间的差距在哪，怎么把 AI 快速原型一步步变成生产级代码，环境搭建、测试策略、CI/CD 集成、监控告警的全流程

---

## 你可能遇到过的问题

- **你用 AI 搭了一个功能原型，半小时就做出来了，但上线却花了两周**
- **原型在本地跑得好好的，部署到服务器就各种报错，根本跑不起来**
- **AI 帮你写的代码，你自己都不敢上生产——总觉得哪里不对但说不出来**
- **每次改一个小功能都要手动部署一遍，很浪费时间**
- **代码上线后出了问题你都不知道，直到用户来骂你**

原型是「验证 idea」的阶段，生产是「对用户负责」的阶段。从原型到生产之间的距离，就是你的项目到底靠不靠谱的差距。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 原型跑起来没问题，上线也没问题 | 原型只验证了「理想路径」，生产环境有千奇百怪的边缘情况 |
| AI 写的原型，让它再改改就能上线 | AI 不会主动考虑安全、性能、监控、容灾，这些需要你补 |
| 加测试会拖慢速度，原型不需要 | 没有测试的原型重构时根本不敢动，反而拖慢上线速度 |
| 部署就是放到服务器上跑起来 | 部署包括环境一致、依赖锁定、配置管理、回滚方案 |
| 小项目不需要监控 | 没有监控等于盲飞，用户发现问题永远比你早知道 |

---

## 常见问题快速解决

| 你遇到的问题 | 快速操作 |
|-------------|---------|
| 原型环境跑得好，部署跑不起来 | 用 Docker 打包环境，确保开发/测试/生产环境一致 |
| 不敢改原型代码 | 给核心功能写单元测试，有测试保护才敢重构 |
| 每次部署手动操作很烦 | 配 CI/CD，git push 自动部署到服务器 |
| 上线后出问题发现太晚 | 加错误监控（如 Sentry）+ 健康检查端点 |
| 不知道原型哪些代码能复用 | 写测试找到问题，重构时用测试验证 |

---

## 为什么 AI 原型离生产很远

AI 非常擅长快速生成功能原型：你描述需求，它几分钟就能给出一个可运行的 Demo。但这个 Demo 和可上线的生产代码之间，通常隔着以下差距：

| 维度 | 原型 | 生产 |
|------|------|------|
| 错误处理 | 假设一切正常 | 每步都有 try-catch 和 fallback |
| 配置管理 | 硬编码在代码里 | 环境变量或配置中心管理 |
| 安全性 | 没有鉴权、输入无校验 | 防注入、防越权、加密传输 |
| 性能 | 能处理小数据量 | 能处理预期峰值流量 |
| 可维护性 | 一个文件数百行 | 模块化、有测试 |
| 可观测性 | 全靠 print 调试 | 日志+指标+链路追踪 |
| 部署 | 本地跑一下 | CI/CD 自动部署 |

把原型变成生产代码不是「改写代码」，而是给代码补全这七个维度的工程保障。

## 第一步：环境规范化

AI 原型通常在本地跑、用本地的 Python 版本、本地的数据库。换一台机器就炸。

### Docker 化

写一个 `Dockerfile` 让你的应用可以在任何地方一致运行：

\`\`\`dockerfile
FROM python:3.12-slim

WORKDIR /app

# 先装依赖，利用 Docker 缓存加速构建
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 再拷代码，这样改代码不用重装依赖
COPY . .

CMD ["python", "app.py"]
\`\`\`

### 依赖锁定

AI 原型通常只写一个宽松的 `requirements.txt`。部署时要锁定到具体版本：

\`\`\`bash
# 导出当前环境的精确依赖
pip freeze > requirements-locked.txt
\`\`\`

### 环境配置管理

\`\`\`python
# ❌ 原型写法
DB_HOST = "localhost"
API_KEY = "sk-xxx"

# ✅ 生产写法
import os
from dotenv import load_dotenv

load_dotenv()

DB_HOST = os.getenv("DB_HOST", "localhost")
API_KEY = os.getenv("API_KEY")
if not API_KEY:
    raise ValueError("API_KEY 未设置")
\`\`\`

## 第二步：测试覆盖

没有测试的代码，你不敢重构，不敢上线。对 AI 原型来说，测试更是保命符——因为你不知道 AI 写的代码在哪些边界条件下会崩。

### 测试金字塔（从底到上）

| 层级 | 覆盖什么 | 速度 | 数量 |
|------|---------|------|------|
| Unit Test | 单个函数、单个类的逻辑 | 毫秒级 | 很多 |
| Integration Test | 模块之间的交互、数据库调用 | 秒级 | 中等 |
| E2E Test | 完整用户流程 | 分钟级 | 少量 |

### 快速给 AI 原型补测试

用 AI 写测试是最有效的做法。给 AI 这个 Prompt：

\`\`\`
给我这段代码写 pytest 单元测试，覆盖：
1. 正常路径
2. 空输入
3. 异常输入
4. 边界值

要求测试函数命名有意义，每个测试只测一个点。
\`\`\`

### 核心测试优先级

如果时间有限，按这个顺序补测试：

1. **核心业务逻辑** —— 出错了用户直接受影响
2. **边界条件** —— 空值、超大数据、特殊字符
3. **外部依赖的异常处理** —— API 超时、数据库断连
4. **配置加载** —— 环境变量缺失时报错而非静默失败

## 第三步：CI/CD 集成

CI/CD 的目的是：每次代码变更都能自动验证、自动部署，减少人工操作的失误。

### 基础 CI 流程（GitHub Actions 示例）

在项目根目录创建 `.github/workflows/ci.yml`：

\`\`\`yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Run tests
        run: pytest --cov=./ --cov-report=term-missing
      - name: Lint check
        run: ruff check .
\`\`\`

### 基础 CD 流程

CD 可以简单地用 rsync 或 scp 同步到服务器，也可以是完整的 Docker 部署：

\`\`\`yaml
deploy:
  needs: test
  runs-on: ubuntu-latest
  if: github.ref == 'refs/heads/main'
  steps:
    - name: Deploy to server
      uses: appleboy/ssh-action@v1.0.3
      with:
        host: ${{ secrets.SERVER_HOST }}
        username: ${{ secrets.SERVER_USER }}
        key: ${{ secrets.SERVER_SSH_KEY }}
        script: |
          cd /app/my-project
          git pull
          docker compose up -d --build
\`\`\`

### CI/CD 的门禁

在 AI 开发工作流中，可以设置这些门禁：

1. **PR 门禁**：测试不通过不允许合并
2. **代码质量门禁**：Lint 不通过不允许合并
3. **覆盖率门禁**：新增代码覆盖率低于 80% 警告
4. **部署门禁**：测试通过后自动部署到 Staging，手动确认后再部署到 Production

## 第四步：日志与监控

日志和监控不是上线后才加的，而是写入代码时就该考虑的。

### 用 Logger 代替 Print

\`\`\`python
# ❌ 原型写法
print("User created:", user_id)

# ✅ 生产写法
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

logger.info("User created", extra={"user_id": user_id})
\`\`\`

### 健康检查端点

所有生产服务都应该有一个健康检查端点，让监控系统知道服务是否活着：

\`\`\`python
# Flask 示例
@app.route("/health")
def health():
    return {"status": "ok", "timestamp": time.time()}

@app.route("/ready")
def ready():
    try:
        db.session.execute("SELECT 1")
        return {"status": "ready"}
    except Exception:
        return {"status": "not ready"}, 503
\`\`\`

### 错误监控

用 Sentry 这类工具做实时错误追踪：

\`\`\`python
import sentry_sdk

sentry_sdk.init(
    dsn=os.getenv("SENTRY_DSN"),
    traces_sample_rate=1.0,
)
\`\`\`

### 需要监控的指标

| 指标 | 为什么重要 |
|------|-----------|
| 请求延迟（P50/P95/P99） | 用户实际体验 |
| 错误率 | 代码是否在生产环境崩了 |
| API Token 使用量 | 控制成本 |
| 数据库连接数 | 防止连接池耗尽 |
| 磁盘/内存使用 | 资源规划 |

## 从原型到生产的逐级路径

不是所有项目都要一步到位。按项目的重要程度分级别：

### Level 1：个人工具
- 最简 Docker 打包
- 写核心功能的测试
- `.env` 管理配置

### Level 2：团队内部工具
- 加 CI（测试自动跑）
- 集成 Sentry 错误监控
- Health check 端点

### Level 3：对外服务
- 完整 CI/CD 流水线
- 性能测试和压力测试
- 日志聚合（如 ELK）
- 告警通知（钉钉/企微/Slack）
- 回滚方案

---

## 你现在应该做什么？

1. 检查你最近的 AI 原型项目：有没有 .env 文件管理配置？依赖有没有锁定？
2. 写一个 `pytest` 测试覆盖核心功能，确认它能在 CI 里跑起来
3. 配一个最简单的 GitHub Actions CI，push 代码自动跑测试
4. 加一个 `/health` 端点，集成 Sentry

---

**要点总结**
- AI 原型和可上线代码之间隔着七个维度：错误处理、配置、安全、性能、可维护性、可观测性、部署
- 环境规范化第一步：Docker + 依赖锁定 + 环境变量管理
- 测试是上线的保命符，按核心逻辑、边界条件、外部依赖、配置的顺序补测试
- CI/CD 自动化验证和部署，设置测试门禁确保代码质量
- 日志、健康检查、错误监控是生产环境的标配，写代码时就该加上
- 按项目重要程度分级别投入，个人工具不用照搬大厂标准
