# 06-实验设计与评估方法

> 读完本文你将了解：不同任务该选什么 metric、怎么设 baseline、Ablation study 怎么做、random seed 管理策略，以及统计显著性检验该不该用。

---

## 你大概率遇到过的问题

做实验时经常碰到：

- **不知道用什么 metric 评估模型**——分类任务用 accuracy 还是 F1？生成任务用 BLEU 还是 perplexity？
- **baseline 选得太随意**，随便选了个 old method 一比就发 paper，审稿人直接质疑
- **结果比别人好零点几个点，但不确定是不是随机波动**
- **做 ablation study 不知道应该去掉什么**，或者做了也不知道怎么看结果
- **换个 random seed 结果差很大**，不换 seed 又担心 reviewer 质疑
- **实验做完了 reviewer 问「significant test 做了吗」**，你从来没做过

这些问题归根结底是：**实验设计不只是跑个结果，而是让你的结果经得起推敲**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| Accuracy 最高就是最好的模型 | 在 imbalanced 数据上 accuracy 会骗你 |
| 一篇论文里 accuracy 高就是好 paper | Reviewer 更看重实验的严谨性，不只看绝对值 |
| Ablation study 就是去掉一个模块看掉多少 | 好的 ablation 是系统性地验证每个设计选择的价值 |
| 多个 random seed 跑一次取平均就行 | 应该报告 mean + std，并且种子数不能太少（至少 3-5 个） |

## Metrics 选择指南

不同的任务需要不同的 metric。选错 metric 是实验设计最常见的问题。

### 分类任务

| Metric | 适用场景 | 注意 |
|--------|---------|------|
| Accuracy | 类别平衡、错误代价相近 | 不平衡数据上会被多数类主导 |
| F1 (macro / micro / weighted) | 类别不平衡 | macro 对 minority class 敏感，micro 受 majority class 影响 |
| AUC-ROC / AUC-PR | 二分类、排序问题 | AUC-PR 比 AUC-ROC 更适合不平衡数据 |
| MCC (Matthews Correlation) | 二分类、不平衡数据 | 被认为是最 balanced 的指标 |

### 生成任务

| Metric | 适用场景 | 注意 |
|--------|---------|------|
| BLEU | 机器翻译 | n-gram 匹配，和人类判断相关性一般 |
| ROUGE | 摘要生成 | 基于召回率，和 BLEU 互补 |
| Perplexity | 语言模型 | 越低越好，但低 perplexity 不等于生成质量好 |
| BERTScore | 任意生成任务 | 基于 contextual embeddings，和人类判断相关性更高 |

### 评估的黄金原则

**Report multiple metrics.** 每个 metric 都有 bias，用两个以上互相补充。

## Baseline 怎么选

### 一个合格的 baseline set 包含

1. **Naive baseline**：最简单的规则或传统方法（如 majority classifier、TF-IDF + SVM），证明你任务本身的难度
2. **Strong baseline**：当前领域常用的最新方法（如你的前一篇 SOTA），证明你比现有最好方法好
3. **Ablated version of your method**：你的方法去掉关键组件后的结果，证明每个组件都不是多余的设计

### Baseline 来源

- 你领域最近 2-3 篇顶会论文的实验部分
- Leaderboard（如 PapersWithCode、GLUE 榜单）
- **不要只选对你有利的 baseline**，Reviewer 会看出来

```python
# 你的实验结果 table 至少要有这些行
results = {
    "Majority Class": 0.52,
    "SVM (BoW)": 0.67,
    "BERT-base": 0.82,
    "Ours (w/o attention)": 0.84,
    "Ours (full)": 0.87,  # your method
}
```

## Ablation Study 怎么做

Ablation study 的目的是：**证明你方法中的每个设计选择都有用**。

### 标准做法

1. 从 full model 开始
2. 每次去掉一个组件（或者替换成一个简单版本）
3. 观察性能变化
4. 变化大的组件 -> 贡献最大

```python
ablations = {
    "Full model": 0.87,
    "w/o attention": 0.84,       # -3%, 注意力机制有用
    "w/o residual": 0.80,        # -7%, 残差连接很重要
    "w/ simple pooling": 0.83,   # -4%, 你的 pooling 方法比简单的强
    "w/o data augmentation": 0.85,  # -2%, 数据增强对效果有贡献
}
```

### 进阶 ablation

- **参数敏感性分析**：改变核心超参数，看方法稳定与否
- **数据量 abalation**：用不同比例的训练数据看增益是否持续
- **跨数据集验证**：在你的方法在 3+ 不同数据集上都有提升才算 robust

## Random Seed 管理

### 为什么需要 multi-seed

PyTorch 的 GPU 操作有非确定性（non-deterministic），同一个 seed 在不同 GPU 上可能结果不同。

而且 single-seed 实验的问题是：**你无法判断性能提升是运气还是方法本身好**。

### 标准做法

```python
# 固定所有随机源
import torch
import numpy as np
import random

seed = 42
torch.manual_seed(seed)
torch.cuda.manual_seed_all(seed)  # multi-GPU
np.random.seed(seed)
random.seed(seed)

# 让 PyTorch 变得 deterministic（可能降低性能）
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
```

用 3-5 个不同 seed 跑：

```python
results = []
for seed in [42, 123, 456, 789, 101112]:
    result = run_experiment(seed=seed)
    results.append(result)

print(f"Mean: {np.mean(results):.4f} +/- {np.std(results):.4f}")
```

## 统计显著性检验

### 什么时候需要

- 你的方法比 baseline 高但差距很小（如 0.5%）
- Reviewer 可能会质疑差异是否来自随机因素
- 你想在论文里写 "statistically significant improvement"

### 怎么做

最简单的做法是 **bootstrapping**：多次随机采样评估数据，比较方法间的差异分布。

```python
import numpy as np

def bootstrap_significance(
    model_preds, baseline_preds, labels,
    metric_fn, n_bootstrap=1000
):
    """Bootstrap test for performance difference."""
    n = len(labels)
    diffs = []
    for _ in range(n_bootstrap):
        idx = np.random.choice(n, n, replace=True)
        score_model = metric_fn(labels[idx], model_preds[idx])
        score_baseline = metric_fn(labels[idx], baseline_preds[idx])
        diffs.append(score_model - score_baseline)

    # p-value: diff <= 0 的比例
    p = np.mean(np.array(diffs) <= 0)
    return p  # p < 0.05 说明显著
```

## 实验设计的常见陷阱

| 陷阱 | 表现 | 如何避免 |
|------|------|----------|
| Data leakage | 验证集数据出现在训练集中 | 严格执行 train/val/test 分离 |
| Metric hacking | 针对特定 metric 过拟合 | Report 多个 metric，不看一个 |
| Cherry-picking | 只报告最好的 seed | Report mean + std |
| Incomplete ablation | 验证了方法在但没验证为什么有效 | 系统性地去掉每个组件 |
| Insufficient baselines | 只和最弱的 baseline 比 | 至少包含 naive + strong + ablated |

---

## 要点总结

1. **Metric 选错 = 实验白做**，分类看 F1/MCC，生成看 BERTScore + ROUGE/BLEU
2. **Baseline 至少三类**：naive baseline、strong baseline、自己的 ablation version
3. **Ablation study 的核心**：逐一去掉一个组件，看性能变化，证明每个设计都有用
4. **Random seed 必须管理**：固定 seed + 跑 3-5 次报告 mean +/- std
5. **差距小时做显著性检验**：bootstrapping 是最简单的方法
6. **避免常见陷阱**：data leakage、metric hacking、cherry-picking 是审稿人最爱抓的问题
