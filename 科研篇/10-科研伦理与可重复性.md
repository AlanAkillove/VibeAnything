# 10-科研伦理与可重复性

> 读完本文你将了解：数据隐私的注意事项、引用规范、AI 生成内容的署名问题、Reproducibility Checklist 的完整内容、以及科研中什么可以做、什么绝对不能做。

---

## 你大概率遇到过的问题

科研伦理听起来很远，但实际操作中每个环节都可能踩雷：

- **用了 GitHub 上的数据集，没看 license**，发表后发现该数据集是非学术用途的
- **引用格式不规范**，直接引用了论文 A 里引用论文 B 的描述——这叫二次引用，写论文时不可以这么写
- **用 AI 润色了论文，被导师问用了没**，不知道怎么回答
- **想做可复现的研究，但不知道 checklist 具体有什么**
- **实验代码公开了但数据集有隐私问题**，不知道怎么处理

这些问题不是小概率事件——**科研伦理和可复现性是现代科研的基本素养，不是锦上添花**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 科研伦理只涉及人体实验和动物实验 | 数据隐私、引用规范、AI 署名、开源许可都是科研伦理的一部分 |
| 引用别人的论文不会出错 | 二次引用、不当引用是常见的学术不规范行为 |
| 用 AI 改写论文不用声明 | 越来越多的会议和期刊要求声明 AI 使用情况 |
| 数据开源就是放在网上 | 开源数据涉及隐私、许可、伦理审查等多方面考量 |
| Reproducibility 是别人复现不了的问题 | Reproducibility 是每个研究者对自己工作的基本要求 |

## 数据隐私

### 你该关心的数据隐私场景

**场景 1：使用公开数据集**

不是所有公开数据集都能用。先看 license：

- **MIT / Apache 2.0 / CC-BY**：通常可以自由使用
- **CC-BY-NC**：限制非商业用途——这就困住了，发表论文是学术用途（算非商业，但边界模糊）
- **CC-BY-SA**：要求衍生作品使用相同许可证
- **无 license**：默认情况下不授权使用——不要用没有 license 的数据集

**场景 2：收集新数据**

如果自己做实验需要收集数据，尤其涉及人类参与者：

- 用户的聊天记录、评论、照片都可能包含个人身份信息（PII）
- **收集前做匿名化**：去除姓名、邮箱、IP、设备 ID 等
- **获得知情同意**：如果有人的数据被收集，需要告知用途并征得同意

**场景 3：发布包含隐私数据的代码**

```python
# ❌ 不要这样做
data = pd.read_csv("user_chat_logs.csv")
model.train(data)

# ✅ 这样做：数据不纳入 repo，只放示例数据
data = load_dummy_data()  # 真实数据在 encrypted 存储
```

## 引用规范

### 基本原则

- **必须引用原始来源**，不能引用「引用中的引用」
- **错误示范**：论文 A 说「Transformer outperformed RNNs (Vaswani et al., 2017)」，你写论文时引用论文 A 而不是 Vaswani 2017

### 什么是正确的二次引用

```latex
\cite{vaswani2017attention}  % 引用原始来源
% 而不是
\cite{paperA_that_cites_vaswani}  % 错误，这是二次引用
```

### 什么时候应该引用

- 使用他人的 idea、方法、代码、数据集时
- background 知识来自特定来源（不是常识）
- 自己和已有工作的差异需要明确说明时

### 什么不需要引用

- 领域常识（如「Transformer 是 2017 年提出的」——需要，这不算常识）
- 自己的方法（引用你自己的 prior work）

## AI 生成内容的署名问题

### 核心原则

**AI 不能是作者。**

- Nature、Science、ICML、NeurIPS 等顶会/顶刊已明确：AI 工具不能列为作者
- 作者必须是对论文有实质性贡献且能为论文负责任的个人

### AI 使用声明

不同 venue 的要求不同，但趋势是**强制声明**：

```
# 最常见的中性声明
The authors used GPT-4 for language polishing and code generation during the
writing process. All generated content was reviewed and verified by the authors.
```

### 什么场景下需要声明

| 使用场景 | 是否需要声明 | 建议措辞 |
|---------|-------------|---------|
| 语法润色 | 通常不需要 | 但部分 venue 已要求声明 |
| AI 生成代码 | 可以声明，也可不声明 | 等同于使用了 code assistant |
| AI 生成论文初稿 | 强烈建议声明 | 在你的贡献和 AI 的贡献间做区分 |
| AI 生成的核心 idea | 这本身就有问题 | idea 应该由人产生 |

## Reproducibility Checklist（完整版）

以下是一个完整的可复现性检查清单，基于 NeurIPS 的 Reproduction Checklist 改编：

### 代码和数据

| 检查项 | 说明 |
|--------|------|
| 代码是否公开在 GitHub / Zenodo | 包括训练和评估代码 |
| 是否有完整的 README（环境配置 + 运行步骤） | 让有基础的人 1 小时内能跑通 |
| 是否有 environment.yml 或 requirements.txt | 精确的版本号 |
| 数据集是否公开 | 或提供获取方式 |
| 预处理步骤是否可复现 | 在代码中写明，或提供预处理后的数据 |

### 实验设置

| 检查项 | 说明 |
|--------|------|
| 所有超参数是否列出 | 包括你改过的和没改过的默认参数 |
| Random seed 是否固定 | 报告用了哪些 seed |
| 是否报告了 mean +/- std | 至少 3-5 个 seed |
| Hardware 配置是否说明 | GPU 型号、显存、CPU、RAM |
| 训练/推理时间是否报告 | 多少小时一个实验 |

### 评估

| 检查项 | 说明 |
|--------|------|
| 评估指标的定义是否清楚 | 公式或引用来源 |
| Baseline 代码是否公开或可用 | 不能自己实现一个有偏的 baseline |
| 统计显著性检验是否做了 | 如果有必要的话 |

## 什么可以做、什么不能做

### Green zone（完全没问题）

- 用 AI 润色语言、修正语法
- 用 AI 生成实验代码和可视化代码
- 用 AI 辅助生成论文的 Initial draft（你提供了完整的大纲和技术细节）
- 在致谢或声明中说明 AI 使用情况
- 用公开许可证的数据集
- 公开代码和数据

### Yellow zone（要小心）

- 在未发布数据上训练模型（确保伦理审查和用户知情同意）
- 使用 CC-BY-NC/SA 数据集（确认是否适用于你的用途）
- 使用 AI 生成论文的摘要/引言初稿（要在修改后确保准确性并声明）
- 合作者间共同使用 AI 工具（统一透明规则）

### Red zone（绝对不能做）

- AI 直接代写整篇论文并投稿——这是学术不端
- 编造 / 捏造实验数据——这是最严重的学术道德问题
- 不引用原始来源，二次引用不查原文——学术不严谨，严重可算不当引用
- 用 AI 生成 peer review 意见——已有多起案例被踢出程序委员会
- 不开源代码但声称 reproducible——虚假宣称
- 隐藏对自己不利的实验结果——选择性报告，是学术不端

## 要点总结

1. **数据隐私从数据来源开始把关**：检查 dataset license，不收集未授权的个人数据
2. **引用必须引用原始论文**，不能引用二次来源；引用的地方不能省略
3. **AI 不能是作者**，使用 AI 辅助建议在文中声明，具体措辞根据 venue 要求调整
4. **Reproducibility Checklist 应该从实验第一天开始跟着做**，不是写论文时才想起
5. **Green zone 的内容放手做**（润色、代码生成、公开代码），**Red zone 的内容千万不能碰**（代写、伪造、选择性报告）
6. **科研伦理不是束缚，是你工作的信用背书**——那些被撤稿、被质疑的例子都是从「一点小问题」开始的
