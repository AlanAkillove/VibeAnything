# 01-如何复现他人的项目

> 读完本文你将了解：论文复现的完整流程——从 clone 到跑通再到验证结果，常见的依赖冲突问题及解法，以及 Conda 和 venv 的切换策略。

---

## 你大概率遇到过的问题

看到一篇感兴趣的论文或开源项目，想跑跑看效果，结果：

- 对着 GitHub 页面的 **README** 一顿操作，`pip install -r requirements.txt` 报一堆版本冲突，直接劝退
- **Python 版本不对**，项目要求 3.8 你装了 3.11，某些包根本装不上
- 论文说他们在 A100 上跑的，你只有一张 3060，**显存炸了**不知道从哪减
- 好不容易跑通了，**结果和论文差一大截**，不确定是自己跑错了还是论文水
- 论文公开的 repo 只有 inference 代码，**训练代码压根没放**

这些问题不是你不细心，而是科研项目中「可复现性」本身就是个老大难问题。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| README 上的命令直接复制粘贴就能跑 | README 经常过时，依赖版本可能已经不兼容了 |
| 项目写了 Python 3.8+ 就装最新的 Python | 版本号上限同样重要，3.8+ 不代表 3.12 也能跑 |
| 训练和推理在同一环境就能结果一致 | TensorFlow/PyTorch 的不同小版本间存在精度差异 |
| pip install 报错说明项目写得烂 | 很多时候只是环境差异而已，换 GPU 型号都可能出问题 |

## 复现全流程拆解

### 第一步：读 README 不能只看命令

先扫一眼 README 的几个关键信息：

- **Python version**：明确要求的 Python 版本（3.8/3.9/3.10）
- **Hardware requirements**：推荐 GPU 型号、显存需求（最低 vs 推荐）
- **Installation**：是不是用 conda？有没有写死 CUDA 版本？
- **Checkpoints**：预训练权重在哪里下载？是 HF 直接 pull 还是 Google Drive？
- **Reproducibility**：有没有提供 exact environment (environment.yml / requirements.txt / Dockerfile)？

如果 README 只给了个 `pip install -r requirements.txt` 就完了，那就做好折腾的心理准备。

### 第二步：复现前的环境准备

**永远先用 conda 创建隔离环境。**

```bash
# 创建新环境指定 Python 版本
conda create -n paper-name python=3.9
conda activate paper-name
```

为什么推荐 conda 而不是 venv？

- Conda 能管理 Python 版本和非 Python 依赖（如 CUDA toolkit、cuDNN）
- 很多 ML 项目用 conda 管理最好，因为 **PyTorch/TensorFlow 的 conda channel 版本最稳定**
- 如果项目提供了 `environment.yml`，你只需要：

```bash
conda env create -f environment.yml
```

项目只有 `requirements.txt` 的话，还是建议先 conda create 再 pip install，而不是直接用 venv。

**Conda vs venv 的切换时机：**

| 场景 | 推荐 |
|------|-----|
| 项目有 environment.yml | Conda |
| 项目只有 requirements.txt，纯 Python 依赖 | venv 即可，但用 conda 也行 |
| 需要指定 CUDA 版本 | Conda（可以装 cudatoolkit 特定版本） |
| CI/CD / Docker 部署 | venv 更轻量，配合 Dockerfile |
| 个人实验管理多个项目 | Conda（conda env list 一目了然） |

### 第三步：处理依赖冲突

经典的冲突场景和解决方案：

**场景 1：torch 版本和 CUDA 不匹配**

```bash
# 先查自己的 CUDA 版本
nvidia-smi

# 去 PyTorch 官网找对应版本的安装命令
# https://pytorch.org/get-started/previous-versions/
pip install torch==1.13.1+cu117 --index-url https://download.pytorch.org/whl/cu117
```

**场景 2：requirements.txt 包太多且有冲突**

不要硬装整个 requirements.txt。逐块试：

```bash
# 先装核心框架
pip install torch torchvision torchaudio

# 再装数据处理库
pip install numpy pandas scikit-learn

# 最后装项目特有依赖
pip install -r requirements.txt
```

**场景 3：某个包已经 deprecated 或改名了**

Google 报错信息，看看是改名了（如 `sklearn` -> `scikit-learn`）还是确实不再维护了。

### 第四步：跑通代码

过程管理要做好：

1. **先看代码结构**：data/、models/、utils/、train.py、eval.py，了解整体流程
2. **从 inference 入手**：先跑 eval/demo，确定模型能加载、推理能出结果，再碰训练
3. **减少资源消耗**：显存不够的话，**降低 batch size**、缩短 sequence length、用 fp16/gradient checkpointing
4. **Shapes debug**：常见报错是 `RuntimeError: size mismatch`，检查 model input 和 tensor shape 是否对齐

### 第五步：验证结果是否靠谱

跑通了不等于复现成功。你要验证：

- **最终指标是否和论文接近**（准确率差 1-2% 正常，差 5%+ 说明有问题）
- **Inference 结果是否一致**：相同 input 跑两遍，输出应该一致
- **Loss curve 形态是否相似**：论文通常有 training curve 图，你的 loss 下降趋势应该类似

如果结果对不上，从这几个方向排查：

- 是不是默认超参数和论文不同？
- ImageNet normalization 之类的 preprocessing 是否一致？
- Random seed 有没有固定？（PyTorch 的 `torch.manual_seed(0)` 之类的）
- 是不是 PyTorch 版本差异导致的精度差异？

## Conda 的日常管理技巧

```bash
# 列出所有环境
conda env list

# 导出当前环境
conda env export > environment.yml    # 完整版（含版本号）
conda env export --from-history > environment.yml  # 仅手动装过的包

# 根据环境文件创建
conda env create -f environment.yml

# 删除环境
conda remove -n paper-name --all
```

导出 environment.yml 时，用 `--from-history` 更实用，因为完整版会把所有间接依赖也写进去，换个平台可能根本装不上。

---

## 要点总结

1. **复现第一步是读 README**，确认 Python 版本、GPU 要求、依赖管理方式，再动手
2. **永远在隔离环境里操作**，推荐 conda，能用 environment.yml 就用
3. **依赖冲突的核心解决顺序**：先核心框架 -> 数据处理 -> 项目特有依赖，逐层调试
4. **从 inference 开始验证**，先确定 model 能加载、推理能出结果，再碰训练
5. **结果验证看指标是否接近、inference 是否一致、loss curve 形态是否相似**
6. **导出环境时用 --from-history**，跨平台兼容性更好
