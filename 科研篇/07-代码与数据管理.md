# 07-代码与数据管理

> 读完本文你将了解：如何系统性地记录实验、DVC 做数据版本控制、Weights & Biases 做实验追踪、以及 Reproducibility Checklist 的内容和使用方法。

---

## 你大概率遇到过的问题

实验一多，管理就跟不上了：

- **两个月前的实验完全记不清参数**，当时随便改了下代码，现在不知道跑的什么配置
- **找实验结果是噩梦**——logs 散落在不同文件夹，最终指标要一个一个 script 去翻
- **合作者想复现你的结果**，你发现自己都复现不了——数据预处理脚本找不到了
- **Git 管理超大模型权重和数据集**，push 不上去
- **Reviewer 问你做了多少次实验、每个 seed 的结果**，你只能去一堆文件夹里到处翻

这些问题的根源是：**代码、数据、实验配置、模型权重是四个不同的工件，不能只用 Git 管理所有东西**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| Git 可以管理一切 | Git 不适合管理大文件（数据集、checkpoint），也有推荐大小上限 |
| 实验 log 不需要系统管理 | 没有系统管理的 log 等于没有 log |
| 保存了代码就能复现实验 | 没有数据版本和配置记录，只有代码也复现不了 |
| 结果好的实验才值得记录 | 失败的实验对后续研究同样有价值，知道什么不 work 很重要 |

---

## 实验记录的黄金三要素

每个实验都应该记录这三个东西：

```
1. Code version  —— 你用的哪版代码（Git commit hash）
2. Config —— 所有超参数（lr、batch size、seed 等）
3. Data version —— 数据集的版本（DVC hash 或说明来源）
```

**少一个，复现就不成立。**

## 代码版本：Git

### 基础

每次实验前确保代码有 clean commit：

```bash
git add .
git commit -m "exp: lr 1e-4, batch 32, seed 42"
git tag exp_0315_lr1e4  # 打 tag 方便回查
```

### .gitignore 的正确写法

```gitignore
# 数据集（由 DVC 管理）
data/raw/
data/processed/

# 模型权重（太大）
checkpoints/

# 实验输出
logs/
wandb/
results/

# Python
__pycache__/
*.pyc
.venv/
```

### 后续要看

```bash
# 跑什么实验就记什么
git log --oneline | head -5
git show HEAD --stat  # 看当前 commit 改了哪些文件

# 回溯
git checkout exp_0315_lr1e4
```

## 数据版本：DVC

Git 管代码，**DVC (Data Version Control) 管数据和模型权重**。

### DVC 基础工作流

```bash
# 初始化
dvc init

# 管理数据集
dvc add data/raw/dataset_v3/
# 这会生成 data/raw/dataset_v3.dvc 文件（小文件，可以 commit 到 Git）

# 设置远程存储（支持 S3、GCS、本地目录等）
dvc remote add -d myremote /mnt/storage/dvc-cache/

# push/pull
dvc push
dvc pull

# 切换数据版本
git checkout dataset_v2  # 切换 .dvc 文件
dvc checkout             # 同步实际数据
```

### 什么时候用 DVC

- 数据集有多个版本（迭代标注、清洗、切分）
- 算力和数据存储分离（本地训练，数据在 NAS/云盘）
- 需要精确回溯某个实验的数据版本

### 简单替代方案

如果 DVC 太重，至少做到：

```bash
# 记录数据集来源和 MD5
echo "Dataset: mydata_v3, MD5: $(md5sum data/raw/dataset.tar.gz)" >> experiment_log.txt

# 或干脆在代码里写死数据 URL 和版本号
DATA_URL = "https://storage.example.com/datasets/mydata_v3.tar.gz"
```

## 实验追踪：Weights & Biases

W&B 是目前最流行的 ML 实验追踪工具。

### 核心能力

```
实验仪表盘     -> 一个页面看所有实验
自动记录超参   -> 记录代码里的 yaml 或 dict 配置
自动记录 metric -> loss、acc、learning rate 随 step 变化
对比实验       -> 任意两行实验对比所有参数和曲线
团队协作       -> 整个 lab 共享实验看板
```

### 快速开始

```python
import wandb

# 初始化
wandb.init(
    project="my-project",
    config={                            # 自动记录所有超参
        "learning_rate": 1e-4,
        "batch_size": 32,
        "model": "ResNet-50",
        "epochs": 100,
        "seed": 42,
    }
)

# 训练循环中记录指标
for epoch in range(epochs):
    train_loss = train_one_epoch()
    val_acc = evaluate()

    wandb.log({
        "epoch": epoch,
        "train_loss": train_loss,
        "val_acc": val_acc,
        "learning_rate": scheduler.get_last_lr()[0],
    })

# 结束
wandb.finish()
```

### W&B 的最佳实践

- **总是一个 experiment 一个 run**，不要一个 script 反复改参数手动记录
- **Sweeps**：W&B 的自动超参搜索，支持 grid / random / bayesian
- **Artifacts**：记录模型 checkpoint、数据集版本，端到端可追溯

```python
# 自动记录模型权重
wandb.log_artifact("model.pth", name="model-checkpoint", type="model")
```

## 项目目录结构建议

一个好的实验项目目录长这样：

```
project/
├── data/               # 数据
│   ├── raw/            # 原始数据（DVC）
│   └── processed/      # 预处理后
├── src/                # 源码
│   ├── train.py
│   ├── model.py
│   └── utils.py
├── config/             # 配置
│   └── config.yaml
├── scripts/            # 实验脚本
│   └── run_exp.sh
├── checkpoints/        # 模型权重（.gitignore）
├── logs/               # 训练 log（.gitignore）
├── results/            # 实验结果（.gitignore）
├── notebooks/          # 分析用
├── .gitignore
├── requirements.txt
└── README.md
```

## Reproducibility Checklist

这是 NeurIPS 等顶会要求作者填写的，但你可以在自己实验室就用。

### 核心检查项

| 检查项 | 你的状态 |
|--------|----------|
| 所有超参数是否在论文/附录中列出 | [ ] 需完成 |
| Random seed 是否固定 | [ ] |
| 代码是否公开，且包含 README | [ ] |
| 训练/评估的 hardware 是否说明 | [ ] |
| 数据集来源和使用条件是否说明 | [ ] |
| 预处理步骤是否可复现 | [ ] |
| 三个 seed 以上的结果是否报告 | [ ] |
| All baseline code 是否可从公开渠道获取 | [ ] |
| 代码中是否包含完整训练和评估流程 | [ ] |

### 最佳实践

在 repo 的 README 里加一个 environment 的表格：

| Component | Specification |
|-----------|---------------|
| Python | 3.10.12 |
| PyTorch | 2.1.0+cu121 |
| GPU | NVIDIA A100 (40GB) |
| CUDA | 12.1 |
| Dataset | mydata_v3 (MD5: a1b2c3...) |
| Random seeds | 42, 123, 456, 789, 101112 |

## 要点总结

1. **实验记录三要素**：Code version (Git hash) + Config (超参数) + Data version (来源/hash)，缺一不可
2. **Git 管理代码、DVC 管理数据和模型权重**，分工明确
3. **Weights & Biases 是最推荐的实验追踪工具**，自动记录所有参数和指标、支持实验对比
4. **项目目录结构要规范化**，把数据、代码、配置、日志、checkpoint 分开
5. **Reproducibility Checklist 可以从实验一开始就跟着做**，而不是等写论文时才想起
6. **失败的实验和成功的实验同样值得记录**，知道什么不 work 是高效率研究的标志
