# 04-常见陷阱与调试技巧

> 读完本文你将了解：AI 编码中最常见的五个陷阱长什么样、怎么在它们出现前就预防、以及每种陷阱的调试和修复方法
> 注：本文不批判 AI 工具——所有工具都会出现这些问题，核心是建立「AI 出错了我知道怎么处理」的能力。

---

## 你可能遇到过的问题

用 AI 写代码几个月后，你大概率遇到这些状况：

- **AI 突然改了一个看起来对、实际上错的东西**——比如把 `===` 改成 `==`，程序表面还能跑，但逻辑已经歪了
- **一轮对话太长之后，AI 开始胡言乱语**——让它改一个小功能，它把整个文件结构都改了，还不承认
- **AI 生成了一段看起来很靠谱的代码**——实际上那个函数或 API 根本不存在，它自己编的
- **每次让 AI 做类似的事情，它都用同样的模式**——比如一直用 `useState` 而不用 `useReducer`，哪怕状态已经复杂到需要后者了
- **AI 生成的代码里直接硬编码了 API Key 或者数据库密码**

这些不是 AI 工具本身「不行」，而是 AI 编码固有的模式化问题。**知道它们存在、知道怎么发现和修复，才是真正能用好 AI 工具的关键。**

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| AI 改坏的代码我能一眼看出来 | 改坏的地方往往是「看起来对，实际上边界条件错了」——一眼看不出来的那种最危险 |
| 上下文溢出只会发生在长对话里 | 上下文溢出不只是「对话太长」——文件变大、引入新依赖、切换模型，都可能导致上下文感知退化 |
| AI 幻觉只出现在不常用的 API 上 | 常见框架（React、Express、Django）的 API 也有幻觉——AI 会混版本、混参数顺序 |
| 同一个问题 AI 用同样的方案是最好的 | AI 的「模式化」来自训练数据中的统计优势——它给你的是「最常见」的方案，不是「最适合你当前场景」的方案 |
| 安全问题是别人的事，AI 写的代码不会出安全问题 | AI 默认生成的代码没有任何安全防护——不自己加安全审查，就是在给项目埋雷 |

---

## 常见问题快速解决

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| AI 改坏了代码但看起来好像没问题 | commit 后养成 `git diff` 习惯——逐行看改动，AI 经常在不起眼的地方做「看似无害的修改」 |
| 一轮对话太长，AI 开始乱改 | 每完成一个可用功能就 commit 并新建对话——不要贪图方便在一个对话里堆太多任务 |
| AI 生成了不存在的 API 或函数 | 让 AI 解释自己的代码——「这个 fetchUserData 函数是在哪个文件里定义的？给我看它的源码」 |
| AI 反复用同一个低效模式 | 给出明确的约束——「用 useReducer 替代 useState，因为状态更新逻辑有多个依赖」 |
| 代码里出现了硬编码的密钥 | 用 `.env` 管理密钥 + git hook 自动检测密钥提交（如 GitGuardian、truffleHog） |
| AI 删了不该删的文件 | 每次 AI 执行删除操作前**手动确认路径**——不要无脑点「确认」|

## 陷阱一：AI 改坏代码（Regressive Edits）

**现象**：你让 AI「加一个功能」，它在修改过程中不小心把其他正常功能改坏了。改坏的部分通常不会报错，只是逻辑变了。

**典型案例**：
```
你：帮我给 updateUser 函数加一个日志
AI：加了 console.log，同时把 if (user.id === id) 改成了 if (user.id == id)
```

那一个 `==` 改变了类型比较的语义——`null == undefined` 是 true，`null === undefined` 是 false。程序不报错，但逻辑悄悄变了。

**预防方法**：
1. **每次 AI 修改后必看 diff**：`git diff` 或 IDE 的 diff 视图。逐行过，别跳过
2. **用单元测试兜底**：关键函数有测试覆盖，AI 改坏了测试会失败
3. **限定 AI 的修改范围**：明确说「只改这个函数，不改其他行」

**修复方法**：
1. 发现问题后马上 `git checkout -- 文件名` 恢复原始版本
2. 重新给 AI 更精确的指令，限定修改范围
3. 或者用 Codex CLI/Claude Code 的 `/undo` 命令撤销上一次修改

## 陷阱二：上下文溢出（Context Overflow）

**现象**：AI 在对话初期表现很好，但随着对话变长、文件变大，AI 开始「失忆」——忘记之前的约定、结构、甚至功能需求。

**典型案例**：
```
对话第 1 轮：AI 正确理解了你的项目结构是 monorepo
对话第 15 轮：AI 把 monorepo 当成单项目，生成了冲突的依赖和配置
对话第 20 轮：AI 开始输出和项目完全无关的代码
```

**为什么会出现**：大模型的 context window 有限。即使工具声称「超长上下文」，有效利用的部分通常是上下文的前半段和后半段——中间部分可能被「压缩」或「遗忘」。

**预防方法**：
1. **任务粒度控制**：每完成一个可用功能就 commit + 新建对话。一个对话不要超过 10-15 轮
2. **项目文档永远存在**：把项目结构、技术栈、核心约定写在 README 或 `.cursorrules` 中——每次新对话 AI 都能读到
3. **使用外部文档作为引用**：不要只靠对话记忆，用文件作为持久化上下文（Cursor 的 @file、Claude Code 的文件引用）

**修复方法**：
1. 遇到 AI 开始胡说时，结束当前对话
2. 在项目文档中更新到最新状态
3. 开新对话，从最新文档开始

## 陷阱三：AI 幻觉（Hallucination）

**现象**：AI 生成了一段看起来很专业、但实际要么不存在的 API、要么是错误的配置、要么是合理但完全不适用于你场景的代码。

**典型案例**：
```
你：用 Python 的 requests 库发一个 HTTP 请求
AI：import requests 后就回车了，但在 import 后面加了一句：
    response = requests.send(url, method='POST', payload=data)
    
注意：requests 库根本没有 `send` 方法，正确是 `requests.post(url, data=data)`
```

但新开发者很可能看不出这个幻觉，只会疑惑「为什么代码跑不起来？」

**还有更隐蔽的**：
- **虚构 npm 包**：AI 引用了不存在的第三方库，你装了半天装不上
- **混版本 API**：React 17 的生命周期方法和 React 18 的并发模式混在一起
- **自行发明配置语法**：webpack、Vite、Babel 配置中混入了不存在的参数

**预防方法**：
1. **遇到陌生 API 时让 AI 提供文档链接**——如果它给不出来，就很可疑
2. **让 AI 解释它写的代码**——「逐行解释这段代码在做什么」，幻觉代码在解释时会暴露矛盾
3. **优先让 AI 用你已知的库**——不要让 AI 自由选择第三方依赖
4. **建立「信任但验证」的默认心态**——对 AI 输出的每一行都保持「这可能是错的」意识

**修复方法**：
1. 先手动查文档验证可疑的 API
2. 如果确认是幻觉代码，删除后用明确指令重写——「用 axios 而不是 fetch，用 get 方法」
3. 复杂场景下，用不同模型交叉验证——让 Claude 审查 GPT 生成的代码

## 陷阱四：重复模式与思维固化

**现象**：AI 倾向于对类似用户问题给出相同的解决方案，哪怕这个模式已经不适合项目的当前状态。

**典型案例**：
```
故事线：
第 1 周：AI 建议用 useState 管理三个状态变量——正确，当时状态简单
第 4 周：状态变量变成 8 个，相互依赖——AI 仍然用 useState，不肯自动升级为 useReducer
第 8 周：组件已经 400 行了——AI 仍然用 useState，没有拆分组件的意识
```

AI 不会主动告诉你「这个方案现在该换了」。它给你的是「最常见方案」，不是「最适合你当前阶段的方案」。

**预防方法**：
1. **定期重构引导**：明确告诉 AI「这个函数/组件现在多大？该拆分了，给出拆分方案」
2. **模式升级指令**：「目前用了 useState，但状态依赖越来越多。评估一下是否该换 useReducer 或 zustand」
3. **让不同模型做架构评估**：Claude 在「该重构了吗？」这类问题上比 GPT 更主动

**修复方法**：
1. 主动触发 AI 的「重构建议」模式——给它看完整文件，问「如果从零写这个组件，你会怎么设计？」
2. 让 AI 列出当前模式的缺点和替代方案的优点
3. 考虑用 Memory Bank 记录项目的架构演进历史，帮助 AI 理解「为什么现在需要换方案」

## 陷阱五：安全问题

**现象**：AI 生成的代码默认没有任何安全防护——它考虑的是「能跑通」，不是「安全地跑」。

**常见安全问题清单**：
- **硬编码密钥**：API Key、数据库密码、JWT Secret 直接写在代码里
- **注入漏洞**：SQL 拼接（`f"SELECT * FROM users WHERE id = {user_id}"`）、命令注入
- **XSS**：Markdown 渲染不做内容过滤，直接注入 HTML
- **路径遍历**：用户输入直接拼接到文件路径中
- **敏感信息泄露**：错误信息中暴露堆栈、数据库结构、文件路径

**预防方法**：
1. **给 AI 安全约束**：在系统 Prompt 或 `.cursorrules` 中加一句「所有用户输入必须做 sanitization」
2. **安全审查是独立环节**：完成功能后，专门让 AI 只做安全审查——不要和功能开发混在一起
3. **使用自动化工具**：在 CI 中加入安全扫描（Snyk、CodeQL、GitGuardian）

**修复方法**：
1. 发现密钥硬编码 → 立即撤销 commit + 更换密钥（不是删除，是换新）
2. 发现 SQL 拼接 → 用参数化查询（`cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))`）
3. 发现 XSS → 使用安全渲染库（DOMPurify、escape-html）
4. 每修复一次，在 `.cursorrules` 中加一条规则，防止同一问题重现

## 你现在应该做什么？

1. 打开最近一个 AI 帮你写的项目，运行 `git log --oneline` 看提交历史，找一次 AI 修改后你直接提交的 commit，用 `git diff` 看看那次提交有没有"悄悄改坏"的地方
2. 检查你的项目里有没有硬编码的密钥、密码、API Key——用 `grep -r "sk-\|api_key\|password" .` 扫一遍，发现就立刻迁移到 `.env` 文件
3. 在你的 `.cursorrules` 或系统 Prompt 中加入安全约束：要求所有用户输入做验证、数据库操作用参数化查询、敏感信息读环境变量
4. 下一次 AI 对话超过 15 轮时，主动停下来：让 AI 生成状态快照，开新对话继续——体验"短对话"和"长对话"的输出质量差异
5. 找一个 AI 生成的代码段，故意问它"这段代码在什么边界条件下会出错？"，测试它是否能主动指出自己的盲区

---

**要点总结**

1. **AI 改坏代码是最常见的陷阱**：看起来对的代码悄悄坏了——解决办法是「每次修改看 diff + 关键函数有测试」
2. **上下文溢出是对话管理问题**：不是 AI 不行，是你对话太长了——每完成一个小功能就 commit 开新对话
3. **AI 幻觉不能完全消除但可以降低**：交叉验证 + 让 AI 解释代码 + 查官方文档——三步缺一不可
4. **AI 的模式化是双刃剑**：它让你开发快，但也让你容易陷入「最佳实践误区」——定期主动问 AI「这个方案还合适吗？」
5. **安全是 AI 代码的默认短板**：AI 不会主动处理安全问题——把安全审查做成开发流程的一个独立阶段
6. **所有陷阱的通用解法**：检查（diff）、测试（unit test）、隔离（commit+新对话）、交叉验证（不同模型）
