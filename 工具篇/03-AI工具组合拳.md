# 03-AI工具组合拳

> 读完本文你将了解：为什么要组合多个 AI 工具来做一个项目、如何用 AI 设计 → 用另一个 AI 实现 → 用第三个 AI 审查的完整流程、以及不同工具组合模式的实战对比
> 注：本文的核心理念是「让每个工具做它最擅长的事」，不捆绑在任何特定工具上。

---

## 你可能遇到过的问题

你可能已经在用一两个 AI 工具写代码了，但总感觉哪里不对劲：

- **一个工具从设计到实现到测试全包**——结果后面越改越乱，每个文件质量都不稳定
- **用同一个工具写了需求分析和代码实现**——AI 在「自己设计的方案上自己写代码」，缺乏客观性，出了 bug 很难自己发现
- **只用一个工具做审查**——发现的问题基本都是「建议加注释」这类表面意见，真正的逻辑漏洞一个没抓到
- **换了个工具处理同一个任务**——发现第二个工具给出的方案和第一个完全不一样，不知道该信哪个
- **试过多个工具但不知道怎么配合**——每次切换工具时上下文要重复一遍，反而效率更低

核心认知：**不同 AI 模型有不同的能力特点，让一个工具从头包到尾，相当于让设计师去写底层代码——能做，但不是最优解。**

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 好工具应该能搞定一切 | 每个工具和模型都有「擅长」和「不擅长」——利用差异而不是抹平差异，才是高效用法 |
| 用多个工具是浪费时间，一个够用了 | 一个工具写到底容易陷入「模式化输出」——不同工具互补能显著提高代码质量和方案多样性 |
| 切换工具就要把上下文重新讲一遍 | 项目文档和 Memory Bank 是跨工具的「共享上下文」——写好文档，切换工具无需重复 |
| 审查阶段用同样的模型就够了 | 同一个模型很难跳出自己的思维定式——用不同模型审查自己没发现的问题，是性价比最高的做法 |
| 组合拳太复杂，只有团队才能用 | 单人开发同样可以——甚至因为没有人替你交叉检查，组合拳的价值反而更大 |

---

## 常见问题快速解决

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 不知道哪个工具适合哪个阶段 | 设计阶段用侧重推理的模型（如 Claude），实现阶段用侧重速度和代码生成的模型（如 GPT），审查阶段用另一个模型交叉验证 |
| 切换工具时上下文丢失 | 把项目文档、架构决策、技术选型写到一个共享文档里——项目 README 或 Memory Bank |
| 一个工具写出 bug，同工具审查不出来 | 换一个完全不同厂商的模型来审查——它能用不同的「思维模式」发现遗漏问题 |
| 觉得组合拳效率低 | 把「设计-实现-审查」做成固定工作流，每次按流程走——熟练后效率反而比单工具高 |
| 不知道审查到底在看什么 | 重点审查三样：逻辑漏洞（边界条件没处理）、安全风险（SQL 注入、XSS）、性能问题（N+1 查询、不必要的循环） |

---

## 核心理念：给每个阶段选对模型

不同 AI 模型的能力分布不是均匀的：

| 阶段 | 推荐模型类型 | 为什么 |
|------|-------------|--------|
| **设计/架构** | 侧重推理和全局观的模型（Claude, Gemini） | 架构设计需要「统筹全局」，发现隐含依赖和边界情况——推理深度比代码量更重要 |
| **实现/编码** | 侧重速度和代码生成的模型（GPT, Kimi, Qwen） | 实现阶段需要效率和代码覆盖率——生成快、能跑通是第一优先级 |
| **审查/测试** | 和前两个不同的模型厂商 | 交叉验证需要「没写过这段代码的眼光」——同一个模型很难发现自己的盲区 |

## 三条实战路径

### 路径一：设计-实现-审查三阶段分离

这是最通用、性价比最高的模式，适合大多数项目。

**场景**：做一个 AI 笔记助手 Web App

**Step 1: 设计（用 Claude）**
```
Prompt: "我想做一个 AI 笔记助手 App，核心功能是：
1. 支持 Markdown 编辑和实时预览
2. 用 AI 自动对笔记做摘要和标签分类
3. 支持全文搜索
请帮我做架构设计，包括技术栈推荐、数据模型、API 设计、前端组件树"
```

Claude 的深度推理会给你一个结构清晰、边界意识强的设计文档。它会主动问：「搜索走什么引擎？」「标签是用户定义还是 AI 自动？」「要不要支持多人协作？」——这些问题在设计阶段解决，比开发时才发现便宜 10 倍。

**产出物**：一份架构设计文档+API 接口定义+数据模型 Schema

**Step 2: 实现（用 GPT 或 Kimi Code CLI）**
```
基于上一步的设计文档，用 Next.js + SQLite 实现
优先做：Markdown 编辑器组件、笔记 CRUD API、全文搜索功能
```

GPT 的代码生成速度快，对常见框架（React, Next.js, Express）的处理很熟练。它会按照设计文档逐块生成代码。

**产出物**：可运行的代码，包含基本功能

**Step 3: 审查（用 Claude Code 或审查专用的不同模型）**
```
审查这个笔记 App 的完整代码：
1. 安全漏洞（XSS、SQL 注入、文件路径遍历）
2. 边界条件（空笔记、超长标题、并发写入）
3. 性能隐患（不必要的重渲染、N+1 查询）
4. 代码风格一致性
```

因为 Claude 没写过这段代码，它的「眼光」是新鲜的——很容易发现 GPT 生成代码中的常见模式化错误。

**产出物**：问题列表+修改建议

### 路径二：同工具多模型切换

如果你只想用 Cursor 一个工具，可以在 Cursor 内部切换不同模型来实现类似效果。

**场景**：用 Cursor 开发一个 Python 数据处理工具

```
Step 1：在 Cursor Chat 中选择 Claude 3.5 Sonnet 模型
→ "帮我设计这个数据处理 Pipeline 的整体架构"

Step 2：切到 GPT-4o 模型继续
→ "按上一步的架构图，实现 data_loader.py、transformer.py、exporter.py"

Step 3：切回 Claude 模型
→ "审查刚才生成的三个文件，找逻辑漏洞和边界情况"
```

**不需要退出编辑器，不需要复制上下文**。Cursor 已经帮你记住了前面对话的内容。

### 路径三：AI 生成测试 → 人类跑 → AI 修复

这是一种高效的「人机协作」循环，特别适合学习：

```
Step 1（Kimi Code CLI）："帮我生成这个函数的完整测试用例"
Step 2（人类）：运行测试，记录失败项
Step 3（Codex CLI）："测试失败了，报错信息是 XXX，帮我修复"
Step 4（Cursor Chat）："解释一下刚才修复的原理"
```

这个循环中，AI 做了「生成」和「修复」，人类做了「决策」和「验证」——各自的优势都用上了。

## 一个完整案例：搭建个人博客系统

用组合拳搭建一个「AI 辅助的个人博客」：

**Phase 1: 架构设计 → Claude**
```
需求：静态博客+AI 自动生成文章摘要和标签+评论区（走 GitHub Issues）
设计输出：技术栈（Astro/Next.js）、组件树、数据流、AI 集成方案
```

**Phase 2: 核心开发 → Cursor（GPT-4o）**
```
按照设计文档实现：
1. 博客页面布局和 Markdown 渲染
2. AI 摘要功能（调 OpenAI API）
3. GitHub Issues 评论区集成
```

**Phase 3: 审查和优化 → Claude Code**
```
代码审查：
- SEO 优化（meta 标签、Open Graph）
- 性能（图片懒加载、代码分割）
- 安全（Markdown 渲染的 XSS 防护）
```

**Phase 4: 内容和管理 → Codex CLI**
```
写几个脚本：
1. 批量导入旧文章（从 WordPress 迁移）
2. 检查断开的内部链接
3. 生成 RSS feed
```

## 你现在应该做什么？

1. 做一次"双工具实验"：用 Claude/ChatGPT 设计一个小功能的架构方案（只设计不写代码），然后把设计文档贴给 Codex CLI/Cursor，让它实现。观察两个工具之间的"接力"是否顺畅
2. 选一个你最近 AI 写的代码，换一个不同厂商的模型做审查——比如 GPT 写的代码让 Claude 审查，Claude 写的让 DeepSeek 审查。对比审查结果，看是不是发现了你没注意到的问题
3. 为你的当前项目写一个共享文档（README 或 Memory Bank），包含技术栈、核心架构决策、目录结构——下次切换工具时先把这份文档贴进去，验证切换是否顺畅
4. 尝试"路径一"完整流程一次：设计（Claude）→ 实现（Cursor/GPT）→ 审查（不同模型），记录每一步花了多少时间、发现了什么问题
5. 把项目共享文档放在项目根目录，作为跨工具的标准输入——养成每次新对话先让 AI 读这份文档的习惯

---

**要点总结**

1. **设计阶段用推理强的模型**：Claude、Gemini——全局观和边界意识比代码量更重要
2. **实现阶段用生成快的模型**：GPT、Kimi、Qwen——效率和覆盖率优先
3. **审查阶段用没写过这段代码的模型**：交叉验证是发现盲区的关键——同模型审查自己的产出效果很差
4. **项目文档是跨工具共享上下文**：写好 README、API 文档、架构决策记录——切换工具不需要重复沟通
5. **组合拳不增加复杂性，而是分散风险**：每个工具做好自己最擅长的事，整体质量和效率都更高
6. **从最简单的一步开始**：不用一开始就三条路径全用——先从「设计用一个，实现用另一个」开始，体验差异后再加审查环节
