# 02-Token 与上下文窗口

> 读完本文你将了解：什么是 Token、它和字数有什么区别、为什么 LLM 会"忘记"前面说过的话、以及如何在自己的工作中利用这些知识

---

## 你可能遇到过的问题

你在和 LLM 聊天，一开始它表现很好——理解你的需求、给出准确的回答。但随着对话变长：

- **它开始忽略你最早说过的话**——明明开头就交代了背景，现在它好像完全不记得了
- **你贴了一大段代码让它分析，它报错了**——不是因为代码有问题，而是提示词太长
- **计费让你困惑**——为什么同样的字数，中文比英文贵很多？
- **你试图让 AI 读一本书，它说"内容超出最大长度"**

这些问题背后都指向同一个概念：**Token**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| LLM 处理的最小单位是"字"或"词" | LLM 处理的最小单位是 **Token**——它既不是字也不是词，而是介于两者之间的子词单元 |
| 1 个汉字 = 1 个 Token | 1 个汉字 ≈ 1-2 个 Token，但 1 个英文单词 ≈ 1-2 个 Token |
| LLM 能记住整段对话历史 | LLM 只能记住最近固定数量的 Token，超过的部分会被"挤出"上下文窗口 |
| Token 越多越贵，所以应该尽量缩短输入 | Token 确实影响成本，但过分简化的 prompt 可能导致质量下降，有时反而更浪费 |

---

## 遇到这些问题怎么办？

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| AI 忘记对话开头的内容 | 在关键节点把重要信息重新说一遍，或者开辟新对话并把关键上下文压缩后放进去 |
| 想贴长代码但怕超长 | 只贴相关部分的代码 + 文件的目录结构，不要贴整个项目 |
| 想分析一本书 | 拆成章节，分多次发，每次让 AI 输出笔记，最后再汇总 |
| 想知道一段文本有多少 Token | 用 OpenAI 的 Tokenizer 工具（platform.openai.com/tokenizer）在线查看 |
| 中文比英文贵 | 理解这是因为中文每个字需要更多 Token 编码。在写 prompt 时，中英文混合（术语用英文）反而比纯中文省钱 |

---

## 什么是 Token？

**Token 是 LLM 处理文本的最小基本单位。**

它不是"字"也不是"词"，而是介于两者之间的——**subword（子词）**。

### 通过例子理解

在 OpenAI 的编码方案（cl100k_base）中：

```
"hello"          → 1 个 Token
"hello world"    → 2 个 Token
"unbelievable"   → 3 个 Token: ["un", "believe", "able"]
"I am a student" → 4 个 Token
```

```
"你好"           → 2 个 Token
"你好世界"       → 3-4 个 Token
```

关键在于：**英文一个词通常 1-2 个 Token，中文一个字通常 1-2 个 Token——所以表达同样的信息量，中文需要的 Token 数是英文的 2-3 倍。**

> 这不是偏见，是编码方式决定的。LLM 的 tokenizer 是在英文为主的语料上训练的，对中文的分词颗粒度更细。

### Tokenizer 是怎么工作的

Tokenizer（分词器）的工作分两步：

1. **把文本切分成子词单元**——基于统计频率，最常见的词保留完整，少见的词拆成更小的片段
2. **把每个子词映射到一个数字 ID**——LLM 内部处理的不是文字，而是一串数字

```
"我喜欢编程"
    ↓ Tokenizer
["我", "喜欢", "编程"]  或  ["我", "喜", "欢", "编程"]
    ↓ 映射
[2847, 8932, 15678]   ← LLM 实际处理的是这些数字
```

不同模型的 tokenizer 不同，同一个文本在不同模型中的 Token 数量可能不一样。

## 什么是上下文窗口？

**Context Window（上下文窗口）** 是 LLM 在生成下一次回答时能"看到"的最大 Token 数量。

可以把 LLM 想象成一个读了一段文字后需要接着往下写的人。他面前摆着一张纸，纸上最多能写 **N 个字**。新的内容不断加进来，最早的内容就会被挤出去。

### 不同模型的上下文窗口大小

| 模型 | 上下文窗口 |
|------|-----------|
| GPT-4o | 128K tokens |
| Claude 最新 Sonnet | 200K tokens |
| Gemini 2.5 Pro | 1M tokens |
| DeepSeek 最新版 | 1M tokens |
| Kimi 最新版 | 256K tokens |
| Qwen 最新 Coder 版 | 128K tokens |
| Llama 最新版 | 256K tokens |
| GLM 最新版 | 128K tokens |

128K Token ≈ 约 8 万汉字 ≈ 一本 200 页的书

200K Token ≈ 约 13 万汉字

1M Token ≈ 约 65 万汉字 ≈ 整套《三体》的篇幅

### 上下文窗口的组成

每次请求消耗的 Token = **输入（prompt）** + **输出（completion）**

```
输入（你发给 LLM 的内容）：     15,000 tokens
输出（LLM 生成的内容）：        5,000 tokens
────────────────────────────────────
本次请求消耗：                 20,000 tokens
当前对话累计：                 80,000 / 128,000 tokens（窗口的 62.5%）
```

对话越长，上下文越满。当超过窗口上限时，**最早的内容会被丢弃**——这就是为什么 AI 会"忘记"你开头说的话。

## Token 和钱的关系

大多数 API 按 Token 计费。不同模型的价格差异很大：

| 模型 | 输入价格（每百万 Token） | 输出价格 |
|------|------------------------|---------|
| GPT-4o | $2.50 | $10.00 |
| Claude 最新 Sonnet | $3.00 | $15.00 |
| DeepSeek 最新 Flash 版 | ¥0.14 | ¥0.28 |
| Gemini 2.5 Flash | $0.15 | $0.60 |

一个简单的换算：

```
你写了一篇 2000 字的文章让 AI 修改
→ 约 2,000-3,000 Tokens（中文）
→ 在 GPT-4o 上约 ¥0.04~0.05
→ 在 DeepSeek 最新 Flash 版 上约 ¥0.0004

你上传了一个完整的项目（10 个文件，共 5000 行代码）
→ 约 30,000-50,000 Tokens
→ 在 GPT-4o 上约 ¥0.30
→ 在 DeepSeek 最新 Flash 版 上约 ¥0.005
```

对日常使用来说成本很低，但如果你做大量批量任务（比如用 AI 处理几十万条数据），Token 成本就需要认真计算了。

## 你现在应该做什么？

1. 打开 [OpenAI Tokenizer](https://platform.openai.com/tokenizer) 在线工具，输入一段中文和一段英文，比较它们的 Token 数
2. 复制你最近一次和 ChatGPT 的对话，看看一共消耗了多少 Token
3. 在下一次聊天中，留意 AI 什么时候开始"遗忘"你的早期指令——这就是上下文窗口的边界
4. 如果你用 API，在代码里加上 Token 计数，建立成本意识

**要点总结**
- **Token** 是 LLM 处理文本的最小单位，是子词级别（subword），不是字也不是词
- **中文的信息密度更高**，表达同样内容需要 Token 数是英文的 2-3 倍
- **上下文窗口** 是 LLM 一次能看到的 Token 上限，超过上限最早的内容被丢弃
- 不同模型的上下文窗口大小不同（128K ~ 1M），价格也不同
- 理解 Token 机制，能帮你更好地设计 prompt、控制成本、避免"失忆"
