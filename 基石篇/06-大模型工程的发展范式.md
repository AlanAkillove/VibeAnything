# 06-大模型工程的发展范式

> 读完本文你将了解：LLM 应用工程经历了哪几个阶段的发展、每个阶段解决了什么核心问题、你当前处于哪个阶段、以及按什么顺序学习最高效
>
> 注：本文讨论的是**LLM 应用层的工程范式**，不包含模型预训练、微调等算法层工作。

---

## 你可能遇到过的问题

如果你按照这个教程的顺序读下来，已经了解了 LLM 的原理、Token、提示词、模型生态、能力边界。但你可能仍会有这些困惑：

- **这些零散知识到底是什么关系？**——提示词工程和 Agent 有什么区别？RAG 又属于哪个部分？
- **现在模型都有百万上下文了，RAG 还有必要学吗？**
- **概念和工具更新太快**——Function Calling、Memory、Agent、各种框架层出不穷，我该先学什么、后学什么？
- **我今天学的东西明天还适用吗？**——到底该学「怎么用某个工具」，还是学「怎么思考」？

这些问题指向同一个答案：**理解这个领域的工程范式演进**。工具和框架会快速迭代，但底层的问题逻辑和工程范式长期稳定。

---

## 五个阶段的演进

AI 工程化的发展可以概括为五个阶段。**它们不是严格的时间先后关系，而是「关注点」的逐层转移**——后一个阶段不会取代前一个，而是在前一个的基础上叠加，解决上一阶段无法解决的问题。

```
阶段1           阶段2           阶段3           阶段4           阶段5
Prompt       →  Context    →  Agent      →  Harness    →  Loop
Engineering    Engineering    Engineering    Engineering    Engineering
```

## 阶段 1：Prompt Engineering（提示词工程）

### 核心问题

**「怎么让 LLM 输出我想要的结果？」**

这是最基础也是最直接的阶段。LLM 就像一个知识渊博但需要精确指令的新员工，Prompt Engineering 就是学习：怎么跟它说话，它才能准确理解并交付符合预期的结果。

### 关键能力

- 写清晰、无歧义的指令
- 角色设定（Role Prompting）
- 示例驱动（Few-shot）
- 思维链（Chain-of-Thought）
- 迭代优化与错误修正

### 在哪学

已经在 **基石篇/03-提示词工程入门** 中详细讲解。

### 局限

Prompt Engineering 能解决「对话交互」的问题，但解决不了「LLM 本身不知道的信息」——比如你的私有项目文档、实时数据、企业内部知识库。这就自然进入了第二阶段。

---

## 阶段 2：Context Engineering（上下文工程）

### 核心问题

**「怎么给 LLM 提供它需要的背景信息？」**

LLM 的训练数据有截止日期，也不可能记住你项目的所有细节。Context Engineering 关注的是：**如何把外部知识高效、准确地送入 LLM 的上下文窗口**，让它基于你的专属信息作答。

### 关键技术

- **RAG（检索增强生成）**：将文档切分后存入向量数据库，检索相关片段拼接进提示词，是大规模知识库的主流方案
- **持久化记忆（Memory Bank）**：为项目建立跨会话的持久上下文（设计文档、技术栈、进度追踪），让 AI 跨多次对话仍记住项目状态
- **系统提示词（System Prompt）**：在 API 调用中设定固定的角色、规则与输出格式，作为每次请求的基础上下文
- **知识库结构化**：把 FAQ、产品文档、代码库整理为 AI 可高效检索的知识源
- **提示缓存优化**：利用模型的缓存机制复用固定上下文，大幅降低长文本调用成本

### 技术现状（2026 年更新）

2025-2026 年百万级上下文模型普及后，中小体量文档（单本书、单个中小型代码库）已可直接全量送入上下文，无需额外做向量检索。
因此 Context 工程的重心也从「如何把信息塞进去」，转向「如何组织信息结构提升回答质量」与「如何通过提示缓存控制长上下文成本」；RAG 也从「长文本必选项」变为「超大规模知识库（百万级文档）的性能优化方案」。

### 在哪学

进阶篇/02-RAG与知识库、工具篇/05-Memory Bank。

### 局限

Context Engineering 让 LLM「知道」了更多信息，但它仍然是被动的——你需要告诉它每一步做什么。如果任务需要多步推理、主动调用工具、自主决策执行，就需要进入第三阶段。

---

## 阶段 3：Agent Engineering（智能体工程）

### 核心问题

**「怎么让 LLM 自主完成任务，而不是每一步都要我指挥？」**

Agent 的核心是：LLM 作为「大脑」，能理解复杂任务、拆解执行步骤、调用外部工具、并根据反馈自主调整。它不再只是一个聊天机器人，而是一个**能主动行动的执行实体**。

### 关键技术

- **Function Calling / 工具调用**：LLM 可以自主调用外部函数（联网搜索、计算器、数据库查询、代码执行等）
  > 注：Function Calling 本身是模型的原生基础能力，也可用于上下文检索场景，但其核心价值在 Agent 的自主任务执行中充分体现，因此放在本阶段重点讲解。
- **任务分解与规划**：把复杂任务拆成可执行的子任务，按步骤推进
- **自我纠错与异常处理**：Agent 执行过程中遇到错误，能自行判断并调整策略
- **多 Agent 协作**：不同 Agent 承担不同角色（规划者、执行者、审查者），协同完成复杂工作
- **会话内执行记忆**：单任务执行过程中记录中间状态与历史操作，支撑多步流转

### 技术现状（2026 年更新）

随着主流模型普遍原生支持工具调用与深度思考模式，「让模型学会调用工具」已不再是技术难点。当前 Agent 工程的核心挑战，已转向长链路任务的规划稳定性、异常自愈能力、多智能体协作效率等方向。

### 在哪学

进阶篇/01-Function Calling、进阶篇/03-Agent架构入门。

### 局限

Agent 让 LLM「能做更多事」，但原生 Agent 系统非常脆弱——API 超时、错误响应、无限循环、任务跑偏等问题频繁出现，只能做演示 Demo，难以稳定跑在生产环境。这就需要第四阶段来解决。

---

## 阶段 4：Harness Engineering（基础设施工程）

> 也可理解为 LLM 应用的「生产级管控工程」：给 AI 系统套上稳定的「缰绳」，核心目标是把「聊天界面里能用的 Demo」变成「生产环境里可靠运行的产品」。

### 核心问题

**「怎么让基于 LLM 的系统稳定、可控、低成本地持续运行？」**

Harness Engineering 本质是把传统软件工程几十年的成熟实践，适配应用到 AI 系统上，解决可靠性、安全性、成本三大核心问题。

### 关键技术

- **可观测性**：全链路监控 LLM 调用（延迟、Token 消耗、错误率、命中率）
- **重试与降级策略**：API 调用失败的指数退避重试、模型不可用时的降级兜底方案
- **安全与合规**：Prompt 注入防护、敏感数据脱敏、输出内容过滤、合规审计
- **版本管理**：提示词版本控制、模型版本管理、效果 A/B 测试
- **成本控制**：Token 预算管理、提示缓存策略、多模型智能路由，长上下文场景下可降低 90% 以上调用成本

### 在哪学

工作流篇/03-成本意识与Token管理、工作流篇/04-验证闭环。

### 局限

Harness Engineering 让系统「稳定」了，但系统本身不会自动变好——错误不会自动修正，效果不会自动提升，所有优化仍依赖人工手动迭代。这就需要第五阶段的闭环能力。

---

## 阶段 5：Loop Engineering（闭环工程）

> 这里的闭环特指**应用系统层面的效果闭环**，而非模型训练层面的微调/RLHF 训练闭环，核心是让系统在运行中自我迭代优化。

### 核心问题

**「怎么让 AI 系统在运行中持续自我改进？」**

Loop Engineering 关注的是反馈闭环：AI 系统的输出被自动评估，评估结果反过来优化系统本身，形成「运行-评估-优化」的正向循环。这是当前最前沿的领域，也是区分「玩具 Demo」和「成熟 AI 产品」的关键标志。

### 关键技术

- **自动化评估（Evaluation）**：建立自动化评估流水线，量化衡量输出质量、准确率、合规性
- **反馈收集**：用户显式反馈（点赞/点踩）、隐式反馈（用户是否采纳建议、停留时长、修改次数）
- **数据飞轮**：优质反馈数据 → 优化提示词/知识库/微调模型 → 更好的用户体验 → 更多优质数据
- **主动学习**：模型对自己不确定的高价值样本，主动请求人工标注，用最少的标注量获得最大提升

### 在哪学

科研篇/06-实验设计与评估方法、AI协作心法篇/06-会话管理。

## 这些范式对初学者意味着什么？

### 学习优先级建议

如果你是刚接触 LLM 应用开发、零项目经验的入门者，学习优先级建议如下，不必盲目追赶前沿概念：

| 优先级 | 阶段 | 核心理由 | 最小可完成实践目标 |
|--------|------|----------|------------------|
| **P0** | Prompt Engineering | 所有上层应用的基础，适配所有模型 | 写出一份可复用的代码审查/文章润色提示词模板 |
| **P0** | Context Engineering | 做项目立刻就能用上，是绝大多数落地场景的核心 | 基于 RAG 搭建一个个人笔记问答机器人 |
| **P1** | Agent Engineering | 有趣且拓展性强，但需要前两阶段的基础支撑 | 做一个能联网查资料、自动整理摘要的学习助手 |
| **P2** | Harness Engineering | 做真实生产项目时自然需要，提前学容易空泛 | 给你的 Agent 加上调用日志、错误重试和成本统计 |
| **P3** | Loop Engineering | 前沿领域，需要项目与数据积累，适合后期深入 | 给问答机器人加点赞/点踩反馈，统计高频错误并优化提示词 |

> ❌ 常见误区：跳过基础直接学 Agent。很多初学者上来就搭建复杂的 Agent 框架，但连精准的提示词都写不好，最终导致 Agent 执行效果差、错误率高。底层基础越扎实，上层应用的效果上限越高。

### 学范式比学工具更重要

这个领域的变化速度是「以月为单位」的。你学的具体工具（某个 Agent 框架、某个平台的 API）可能半年后就不再主流，但**范式不会变**：理解「为什么需要 Context Engineering」比学会某一个 RAG 框架重要得多。

本教程的各个模块，就是围绕这五个范式组织的：

| 范式阶段 | 教程对应部分 |
|---------|------------|
| Prompt Engineering | 基石篇 03 + AI协作心法篇 |
| Context Engineering | 进阶篇 RAG + 工具篇 Memory Bank |
| Agent Engineering | 进阶篇 Agent + Function Calling |
| Harness Engineering | 工作流篇 + 环境准备篇安全实践 |
| Loop Engineering | 科研篇实验设计 + AI协作心法篇反馈机制 |

这套工程范式完全通用，目前国产主流模型（DeepSeek、Qwen、GLM 等）的工具调用、长上下文、缓存能力均已成熟，国内开发者无需绑定海外模型即可完整实践全阶段内容。

## 你现在应该做什么？

1. 回顾「Prompt → Context → Agent → Harness → Loop」五层范式，判断你目前处于哪个阶段
2. 如果你的目标是「做出一个能用的落地项目」：重点扎实掌握前两个阶段
3. 如果你的目标是「理解这个领域的未来方向」：了解 3-5 阶段的核心逻辑与发展趋势
4. 无论你的目标是什么：**优先学范式原理，而不是死记具体工具**

**要点总结**
- LLM 应用工程按 **Prompt → Context → Agent → Harness → Loop** 五层范式逐层演进，每层解决上一层的核心痛点，是叠加而非替代关系
- 初学者按优先级分步学习，优先打牢提示词与上下文工程的基础，不要跳过基础直接追 Agent 等前沿概念
- 工具和框架会快速迭代，但底层工程范式长期稳定，**优先理解原理，而非死记具体工具**
