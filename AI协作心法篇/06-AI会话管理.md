# 06-AI会话管理

> 读完本文你将了解：什么是会话管理的本质、何时开新会话和何时复用旧会话的判断标准、管理多个会话的表格方法、以及建立属于你自己的会话管理体系
> 注：本文讨论的不是某个 AI 工具的操作界面，而是无论使用什么平台（Claude/ChatGPT/DeepSeek/API 调用）都通用的会话管理原则。会话（session）是你和 AI 协作的最小工作单元，管好会话就管好了你的工作流。

---

## 你可能遇到过的问题

你的 AI 对话列表越来越长，打开任何一个都不知道当时在做什么：

- **你有 20 多个未命名的对话**，每个都叫「新对话」「新对话(3)」「Python 代码」，完全不知道哪个是哪个
- **你想继续一个之前的工作**，但打开对话发现里面混了好几个不相关的任务，你分不清哪些是有效进展、哪些是废弃的尝试
- **你有一个 AI 对话用了一周**，里面堆积了代码、讨论、方案对比、bug 排查好几种类型的任务，每次切换任务都要翻很久历史
- **你想把之前某次对话中的成果拿出来复用**，但发现对话太长，已经找不到那个关键输出了
- **你同时在做 2-3 个项目**，AI 对话列表里混在一起，每次写代码前都要花 5 分钟确认「这对话对应的是哪个项目、讨论到了什么程度」

这些问题不是因为你不够有条理——而是因为你把 AI 会话当作了一个**聊天记录**，而不是一个**工作单元**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 对话列表是自动记录，不需要刻意管理 | 对话列表是你的**项目管理面板**。好的命名和分类让你 10 秒找到需要的内容，差的列表让你每次都要翻 5 分钟 |
| 一个对话里可以混着做不同的事情 | 混入不同目标的对话会产生上下文污染——AI 会在写代码时引用你在另一个话题中提到的错误信息 |
| 关闭的对话就再也用不了了 | 大多数平台支持搜索历史对话。关键的输出应该被**提取出来存到本地**，而不是留在对话历史里吃灰 |
| 开新对话要多写一次背景，麻烦 | 一次背景写的成本（<5 分钟）远低于在一个混乱的旧对话里翻找、排除无关讨论的成本（10-20 分钟） |
| 会话管理是「不必要的仪式感」 | 会话管理不是仪式感，是**工作效率问题**。你花 30 秒命名和分类一个对话，能省下以后找内容的 5 分钟 |

---

## 常见问题快速解决

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 对话列表混乱，找不到目标对话 | 立即做一次清理：为每个未命名的对话加一个前缀标记「项目名-类型-目标」，如「RAG项目-代码-向量检索实现」 |
| 不确定该开新对话还是沿用旧对话 | 简单判断：你接下来要说的话题和当前对话里最后 5 轮讨论的话题是否属于同一个任务？如果不是，开新对话 |
| 想返回之前的对话找一段输出 | 如果你有命名规范：直接搜对话名。如果没有：搜那个输出的关键词，AI 平台的搜索功能通常能找到 |
| 工作中断后回来继续 | 先看对话的前 5 轮和最后 5 轮，确认当时在做什么。如果发现已经混乱了，不要试图「续杯」，开新 session 从总结恢复 |
| 同一个项目多人在用同一个 AI 账号 | 为每个人/每个工作流分配独立的对话线程，命名上标注「人名-任务」，避免互相覆盖上下文 |

## 会话管理的核心原则

### 一个 Session = 一个任务 = 一个输出

这是最核心的原则。每个 session 应该满足：

- 开始时有**明确的目标**
- 过程中只讨论**与目标直接相关**的内容
- 结束时有一个**具体产出物**（代码/方案/笔记/决策）
- 结束后主动**关闭**（不留下一个「还开着但没人记得在做什么」的对话）

### 什么时候开新 Session

按以下标准判断，只要**命中任意一条**就开新 session：

| 条件 | 示例 |
|------|------|
| 目标发生了变化 | 从「写代码」变成了「debug」，这是两个独立的任务 |
| 要讨论一个不同的话题 | 做完 RAG 项目想研究 LangChain 的高级用法，这是不同的知识领域 |
| 要切换项目上下文 | 从项目 A 切换到项目 B，AI 需要完全不同的背景信息 |
| 当前对话已经很长（>30 轮） | 即使任务还没完成，也应该用状态快照压缩后开新 session 继续 |
| 当前对话混入了不相关的内容 | 对话里已经在跑题或者混入了其他任务的讨论，直接开新 session |

### 什么时候可以复用 Existing Session

| 条件 | 示例 |
|------|------|
| 同一个任务的自然延续 | 还在写同一个模块的代码，只是中途被其他事打断了几小时 |
| 任务的逐步推进 | 在做「实现用户注册功能」，写了基础框架，现在继续写验证逻辑 |
| 对同一个问题追问 | 刚讨论了 PostgreSQL 死锁的原理，现在追问「那间隙锁在什么情况下会变成死锁」 |
| 评审反馈迭代 | AI 给出了代码 review，你根据反馈修改后又让它重新 review |

## 具体怎么操作

### 1. 命名规范

#### 模板：[项目名] [任务类型] [具体目标]

| 好的命名 | 差的命名 |
|---------|---------|
| [RAG项目] 代码-向量检索实现 | RAG 代码 |
| [博客] 写作-AI幻觉文章初稿 | 写文章 |
| [学习] 理解-PostgreSQL MVCC | PostgreSQL |
| [Bug] 排查-用户登录302错误 | debug |
| [RAG项目] 设计-知识库Schema | 数据库设计 |

如果你想更精细管理，可以在任务类型上用统一标签：

- [项目] 设计-xxx：设计方案、架构决策
- [项目] 代码-xxx：实现功能、写代码
- [项目] debug-xxx：排查问题
- [项目] review-xxx：审查代码或方案
- [项目] 笔记-xxx：学习、整理知识
- [项目] 写作-xxx：内容产出

#### 群组前缀

对于项目繁多的开发者，按项目分组命名：

`
[RAG] 设计-知识库Schema
[RAG] 代码-向量检索实现
[RAG] debug-embedding维度不匹配
[博客] 写作-AI幻觉文章
[博客] review-文章初稿
[学习] PostgreSQL MVCC
[杂项] 环境配置-DeepSeek API Key
`

这样在对话列表里，所有 [RAG] 开头的对话被自然分组在一起。

### 2. 输出存档

**永远不要依赖 AI 的对话历史作为你的项目文档。** AI 对话是临时的，你的笔记是永久的。

每当你觉得一个 session 完成了某个里程碑，做以下操作：

1. 把关键输出复制到你的项目笔记中（Obsidian/Notion/本地 markdown 均可）
2. 别让 AI summarize——你自己判断什么值得保存。AI 可能会把所有内容都保存，包括废弃的方案和错误尝试
3. 存档的内容包括：
   - 最终产出的代码或方案
   - 关键设计决策（选了什么、为什么、放弃了哪些选项）
   - 遇到的问题和解决方案
   - 下一步计划

这样即使 AI 对话被清除了，你的项目知识体系不会丢失。

### 3. 对话生命周期管理

一个 session 的生命周期应该是：

`
创建 → 命名（目标明确） → 使用（聚焦目标） → 产出 → 存档关键输出 → 关闭
`

定期清理：

- **每天结束**：检查当天的对话列表，把没有命名、不需要保留的对话清理掉
- **每个任务结束时**：将关键输出存档到笔记系统
- **每周一次**：花 5 分钟整理你的对话列表，归档已完成项目的对话

## 跨 Session 工作流

### 同一个任务跨多 Session

当一个任务需要多个 session 来完成时，使用「Session Chain」模式：

`
Session 1: [RAG] 设计-系统架构
  输出：架构文档
  |
Session 2: [RAG] 代码-基础框架搭建
  输出：项目骨架代码
  |
Session 3: [RAG] 代码-向量检索实现
  输出：向量检索模块
  |
Session 4: [RAG] 代码-LLM调用封装
  输出：LLM客户端模块
`

每个 session 之间通过**文档**（不是 AI 对话历史）传递上下文。Session 1 的输出是 Session 2 的输入，Session 2 的输出是 Session 3 的输入。

### 并行 Session

当你需要在多个项目或任务之间切换时，并行 session 是必要的：

`
[RAG] 代码-向量检索实现  ← 当前工作中
[博客] 写作-上下文窗口管理  ← 工作一段时间后切过来写一段
[RAG] 设计-KnowledgeBase Schema  ← 并行设计但还没代码化
`

每个 session 是独立的一个对话，你切换时不会污染彼此的上下文。你在 RAG 项目里不会突然看到博客的内容，反之亦然。

## 参考表格：Session 决策速查卡

| 场景 | 操作 | 理由 |
|------|------|------|
| 刚吃完饭回来，上次对话还开着 | 读最后 3 轮，看看能不能接上 | 如果是同一个任务的直接延续，继续用 |
| 上次对话最后在讨论另一个话题 | 开新 session | 跨话题的上下文会污染 |
| 对话已经 40 轮了但任务还没完 | 状态快照 + 开新 session | 旧对话的上下文已不可靠 |
| 要开始一个新项目 | 开新 session | 不同项目的上下文完全不通用 |
| 要 review 他人代码 | 开新 session，不混入你的项目信息 | 保持评审的独立性 |
| 突然想到一个好点子 | 开新 session 记录，不要在当前对话里插播 | 不打断当前工作流，又不错过新想法 |

## 你现在应该做什么？

1. 打开你的 AI 对话列表，检查每个对话的命名：有多少是描述了目标、多少是默认名？
2. 选 3 个当前最活跃的对话，按照 [项目] 类型-目标 的格式重新命名
3. 在你本地笔记系统中，创建一个「AI 会话日志」文件夹，用于存档每次 session 的关键输出
4. 今天结束时，养成对一个习惯：离开 AI 前，先命名当前对话，再记录关键输出到笔记
5. 复制上面的「Session 决策速查卡」，贴在常用位置，下次犹豫要不要开新 session 时对照

---

**要点总结**
- 一个 session = 一个任务 = 一个输出。**把 session 当作工作单元管理，不是聊天记录**
- 开新 session 的判断标准：目标变化、话题切换、项目切换、对话过长、上下文污染
- 命名规范是性价比最高的管理手段：[项目名] 类型-具体目标 让你 10 秒定位任何对话
- 关键输出必须**存档到本地笔记**，不要依赖 AI 的对话历史——对话会消失、会混乱，你的笔记不会
- 跨 session 之间通过**文档**（不是对话历史）传递上下文，避免上下文污染
