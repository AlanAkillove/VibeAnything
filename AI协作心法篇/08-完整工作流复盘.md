# 08-完整工作流复盘

> 读完本文你将了解：用一套 AI 协作心法的完整框架，在一个完整项目（构建一个 RAG 问答工具）中展示从需求到部署的全过程最佳实践
> 注：本文是一个整合篇，将前 7 篇文章讲述的 AI 协作心法（幻觉识别、意图管理、收敛分析、上下文管理、诚实审查、会话管理、代码审查）放在一个真实的项目场景中运行。它不是理论回顾，而是一个 step-by-step 的操作演示。

---

## 项目概览

**目标**：用 AI 协作的方式，从零构建一个「个人知识库 RAG 问答工具」。用户可以上传 PDF 文档，向文档提问，AI 基于文档内容回答。

**范围**：本地开发环境、Python 后端、命令行界面（CLI），不做 Web UI、不做云端部署。单人开发，周期 3 天。

**AI 协作原则**（来自前 7 篇文章的核心要义）：

1. 每个 session 一个目标，不跑题（#02, #06）
2. AI 的输出预设为「待验证」，关键信息交叉验证（#01）
3. 分析深度可控，不做过度分析（#03）
4. 上下文窗口满了就压缩或开新 session（#04）
5. 让 AI 诚实地指出问题，不 sycophancy（#05）
6. AI 生成的代码必须理解后才能提交（#07）

---

## 第一天：规划和设计

### Session 01：确定功能范围和架构

**命名**：[RAG] 设计-功能范围界定

> Prompt：我想做一个 RAG 问答工具，本地运行，CLI 界面。用户上传 PDF，问问题，AI 基于文档回答。帮我列出 MVP 的功能范围和可选的扩展功能。然后我们确定哪些做哪些不做。

AI 给出了 12 个功能点。我使用**收敛原则（#03）**做了裁剪：

`
AI 列出的功能：
✓ PDF 解析
✓ 文本切分
✓ 向量化存储
✓ 检索
✓ LLM 调用 + 回答
✗ Web UI（不做，保持 CLI）
✗ 多用户支持（不做，单人本地方案）
✗ OAuth 集成（不做）
✗ 批量文档管理（不做，MVP 一次处理少量文档）
✗ 多轮对话记忆（不做，先做单轮 Q&A）
✗ 文档类别分类（不做，MVP 不引入分类体系）
`

**结果**：MVP 明确为 5 个核心功能，放弃 6 个非核心功能。这一裁剪本身避免了**过度分析**的陷阱——我们不会花时间讨论根本不在范围内的东西。

**适用范围控制（#02）**：

> Prompt确认：确认我们的 scope。MVP 只做 5 个功能：PDF 解析 + 文本分块 + 向量化存储 + 检索 + LLM 回答。不做 Web、不做多用户、不做多轮对话。在完成 MVP 之前，我们不讨论任何扩展功能。同意吗？

让 AI 显式确认 scope，建立了一个「契约」。后续 AI 如果提议扩展功能，我可以引用这个契约让它回到正轨。

---

### Session 02：技术选型

**命名**：[RAG] 设计-技术选型

**避免 sycophancy（#05）**——不要让 AI 看着你的预选方案说好话，先让 AI 独立做推荐：

> Prompt：我正在做一个本地 RAG 工具，Python 语言。不考虑部署到服务器。从以下维度给出最佳技术选型方案，每个维度给出你的推荐理由：PDF 解析库、向量数据库、embedding 模型、LLM、文本分块策略。先列方案，不要先问我的倾向。

AI 推荐了 PyMuPDF 做 PDF 解析、ChromaDB 做向量库（轻量、嵌入 Python 进程）、BAAI/bge-large-zh-v1.5 做 Embedding（国内可用: SiliconFlow 提供）、DeepSeek 做 LLM。每个推荐都给了理由。

我选择了一条「不复杂、够用」的技术栈，没有花时间去对比 3 个 PDF 库的性能、没有纠结分块策略的 5 种变体。**够好原则（#03）**应用于选型阶段。

**要点总结（Session 01-02）**：
- 在规划阶段，**会话管理（#06）**开始实施：每个 session 一个目标，命名清晰
- 用**收敛原则（#03）**裁剪 MVP 范围，不做不必要的分析
- 用**反向确认框架（#05）**做了技术选型，优先独立推荐再结合自己的判断
- 用**意图管理（#02）**锁定了 scope，防止规划阶段无限膨胀

---

## 第二天：核心实现

### Session 03：项目骨架搭建

**命名**：[RAG] 代码-项目骨架

从规划进入实现。开一个新 session。开始前，把前两个 session 的关键输出总结成一个**状态快照（#04）**作为新 session 的初始上下文：

> 新 Session 初始上下文：
> 项目：本地 RAG 问答工具，CLI 界面。
> 技术栈：PyMuPDF + ChromaDB + BAAI/bge-large-zh-v1.5（Embedding）+ DeepSeek API。
> MVP 范围：PDF 解析 -> 文本切分 -> 向量化存储 -> 检索 -> LLM 回答。不做 Web UI/多用户/多轮对话。
> 今天的任务：搭建项目骨架，实现 PDF 解析和文本切分两个模块。

> Prompt：基于上述背景，帮我生成项目骨架。按照关注点分离原则，每个模块一个文件，说明每个文件的职责和接口。

AI 生成了目录结构：

`
rag_tool/
├── main.py                 # CLI 入口
├── config.py               # 配置（API Key、模型参数、分块参数）
├── document/               # 文档处理层
│   ├── __init__.py
│   ├── loader.py           # PDF 加载
│   └── splitter.py         # 文本分块
├── vector/                 # 向量存储层
│   ├── __init__.py
│   └── store.py            # ChromaDB 封装
├── llm/                    # LLM 调用层
│   ├── __init__.py
│   └── client.py           # DeepSeek API 封装
├── pipeline/               # RAG 流程编排
│   ├── __init__.py
│   └── rag_pipeline.py     # 完整 RAG 流程
├── requirements.txt
└── README.md
`

**代码审查（#07）**：我让 AI 解释骨架中每个文件的职责（宏观理解），确认没有不合理的耦合（如 splitter.py 不应依赖 ChromaDB）。通过。

---

### Session 04：实现 PDF 解析和文本分块

**命名**：[RAG] 代码-PDF解析和分块

> Prompt：现在实现 document/loader.py：用 PyMuPDF 加载 PDF，返回文本。接口设置为 def load_pdf(file_path: str) -> str。

AI 生成了代码。**审查（#07）**：

- 确认 API 接口是否符合我们之前的设计
- 检查错误处理：文件不存在、PDF 损坏时的行为
- 确认资源管理：文件是否正常关闭

**幻觉检查（#01）**：AI 用了一个 API 参数 page_numbers，我查了一下 PyMuPDF 文档确认存在。通过。

`
💡 这步看起来繁琐，但实际只花了 2 分钟。
   而且让我发现了 AI 假设的一个行为——它用了过于复杂的 page 遍历方式，
   其实 PyMuPDF 可以直接 load 整个文档。
   我要求简化，AI 重写，代码从 15 行变成了 6 行。
   如果不审直接用了，后面维护时就会对着这 15 行代码困惑「为什么这里这么复杂」。
`

**文本分块模块**：

> Prompt：实现 document/splitter.py。分块策略：按段落分块（\n\n），每个 chunk 最大 512 token，重叠 10%。接口：def split_text(text: str) -> list[str]。

AI 实现的版本用了 	iktoken 做精确 token 计数。**理解代码（#07）**：我让 AI 解释了它的分块边界处理逻辑——最后一个 chunk 如果不足 512 怎么处理、重叠部分怎么选择起点。确认逻辑合理后通过。

---

### Session 05：向量化存储和检索

**命名**：[RAG] 代码-向量存储和检索

这是最复杂的部分之一。AI 对 ChromaDB 的封装涉及了多个接口和方法。

**审查要点**：

| 审查维度 | 检查内容 | 结果 |
|---------|---------|------|
| 宏观理解 | 接口是什么？add/query/delete？ | 通过 |
| 资源管理 | ChromaDB 持久化路径在哪？ | 发现默认在内存中，我要求改为持久化到本地目录 |
| 上下文一致性 | embedding 模型和后续检索是否一致？ | 确认一致 |
| 错误处理 | embedding API 调用失败时的 fallback | AI 没处理，要求添加 |
| 边界条件 | 空文档列表、空查询的处理 | AI 没处理，要求添加 |

**诚实审查（#05）**：我没有问「这段代码写得好不好」，而是问「这段代码在哪些边缘场景下会出错」。这让 AI 主动指出了我没有注意到的几个问题——包括 embedding 的 API 调用超时、ChromaDB 的 collection 不存在时的初始化问题。如果没有这个提问框架，AI 很可能就是一句「代码整体不错」了。

---

### Session 06：LLM 调用封装 + RAG Pipeline

**命名**：[RAG] 代码-LLM调用和Pipeline

**诚实测试**：我故意先写了一个草稿方案给 AI：

> Prompt：我打算用简单的字符串拼接把 context 和 question 拼在一起，发给 DeepSeek API。你看看这个方案有什么问题。

AI 给出了一个良好的否定（这部分是因为我在 prompt 里用了「你看看有什么问题」这个反向框架）：

AI 指出：直接拼接可能导致 prompt injection、context 过长超过 API 限制、缺少 system prompt 的 role 设定等 3 个问题。然后给出了一个结构化的 prompt 模板。

**代码理解**：我让 AI 解释 RAG pipeline 的完整流程，从输入问题到输出回答，分步描述。确认每一步的接口、数据流转、错误传播。

**简洁性检查（#07）**：pipeline 的实现中有一段复杂的 prompt 模板字符串格式化。我让 AI 用 f-string 重写，从 10 行变成 3 行，可读性大幅提升。

---

**要点总结（实现阶段 Session 03-06）**：
- 每个模块一个独立 session，session 之间通过**状态快照（#04）**和**代码文件**传递上下文
- 每次 AI 给出的代码都经过 5 步审查：宏观 -> 热点 -> 逐段 -> 验证 -> 精简
- **反向提问框架（#05）**让 AI 主动指出了代码的边缘场景问题
- 对不理解的代码，用「解释-简化-重写」三步走建立了代码所有权
- **幻觉检查（#01）**贯穿始终：每个 API 参数用法都确认了官方文档

## 第三天：整合、测试和部署

### Session 07：集成测试

**命名**：[RAG] 测试-集成测试

> Prompt：基于我们已经实现的所有模块，帮我写一个端到端的集成测试。它应该：1. 加载一个测试 PDF 2. 索引到向量库 3. 执行一次查询 4. 输出结果。测试环境为本地，不需要 mock 外部依赖。

AI 生成了测试脚本。**验证（#01）**：我实际运行了测试。

`
💡 这里遇到了第一个真正的问题。AI 生成的代码中，ChromaDB 的持久化路径配置
   和 document/loader.py 中的路径硬编码不一致。测试跑出来报错。
   原因：前两个 session 中各自配置了不同的存储路径。
   这是多 session 工作流的一个代价——分散决策导致不一致。
   解决：把存储路径统一到 config.py 中，确保所有模块读取同一个配置。
`

这个 bug 后来被修复。**关键教训：跨 session 的配置决策必须记到共享文档中，不能依赖 AI 的上下文记忆。**

---

### Session 08：Bug 修复

**命名**：[RAG] debug-持久化路径不一致

> Prompt：集成测试报错：ChromaDB 持久化路径不一致。loader 用了一个路径，pipeline 用了另一个路径。帮我检查所有文件中配置 ChromaDB 路径的地方，统一到 config.py 中。

AI 找到了 3 个地方。我逐一确认 AI 的修改正确后提交。**代码理解**：改了哪些地方、为什么改、对其他地方有没有影响。

---

### Session 09：性能优化（可选）

**命名**：[RAG] 代码-检索性能优化

MVP 跑通了。实测检索耗时约 0.8 秒。我**时间盒约束（#03）**了性能优化的时间——给 AI 说「我们只有 15 分钟来做这个优化」。

AI 建议了 4 个优化点。根据**优先级排序**，我只做了 P0（设置 ChromaDB 的 
_results 参数限制检索量），其余打上「待观察」标签。

P1-P3 不做——因为当前 0.8 秒的响应时间已经可以接受。这就是过度分析中说的「循环优化陷阱」。

---

### Session 10：代码审查和文档

**命名**：[RAG] review-完整代码审查

> Prompt：以最严格的 code review 标准审查我们整个项目。从逻辑漏洞、安全隐患、代码可维护性三个维度，按严重程度排序出问题列表。先说最严重的问题，不要说总体评价。

AI 给出了 7 个问题（P0-P2），我修复了其中 3 个 P0 和 P1 问题，P2 问题标注为「以后有时间再修」。

**代码所有权检查（#07）**：我通读了项目中的所有代码，对每一段做了一个自检：

- [x] 我能用自己的话描述每段代码做什么
- [x] 我理解每个外部依赖的作用
- [x] 我知道错误处理逻辑
- [x] 我知道边界情况
- [x] 我能独立修改代码
- [x] 如果代码出问题，我能独立 debug

所有项目都打勾。**这行代码真正属于我了。**

---

### 剩余时间：README 和最终提交

让 AI 根据项目文件生成 README 和使用说明。**核实（#01）**：AI 在 README 中写了一个 API 调用示例，我实际跑了一遍确认正确。README 中描述的文件路径和命令行参数，我逐一核对。

最终提交到 Git，打 tag v0.1.0。项目完成。

## 项目复盘

### 数据统计

| 纬度 | 数据 |
|------|------|
| 总 Session 数 | 10 个 |
| AI 协作总时长 | 约 6 小时（分 3 天） |
| AI 生成代码行数 | 约 650 行 |
| 手动修改代码行数 | 约 80 行（配置统一、错误处理增强、测试） |
| 发现的问题数 | 3 个 bug（通过审查发现，非 AI 主动指出） |
| 被拒绝的 AI 建议 | 4 次（过度设计、不必要的优化、范围外功能） |

### 用了哪些心法

| 心法 | 文中应用位置 | 效果 |
|------|------------|------|
| #01 幻觉识别 | API 参数验证、README 核实 | 发现 2 处 API 不一致 |
| #02 意图管理 | 每个 session 一个目标、scope 锁定 | 避免了 Web UI、多用户等不必要讨论 |
| #03 过度分析 | MVP 裁剪、性能优化时间盒 | 每个阶段「够好」就停，不做完美主义 |
| #04 上下文管理 | 状态快照跨 session 传递、session 边界划分 | 长项目拆成 10 个可以闭环的模块 |
| #05 诚实审查 | 反向提问框架、魔鬼代言人模式 | 让 AI 主动指出了路径配置不一致等问题 |
| #06 会话管理 | 命名规范、10 个独立 session | 不会找不到对话、不会互相污染 |
| #07 代码所有权 | 5 步审查法、独立性检查 | 所有代码理解后才提交，技术债务最小化 |

### 最值得保留的习惯

从这个项目中，最值得保留的几个习惯：

1. **每个 session 开头的 scope 确认**——「我们只做 X，不做 Y」。一个 5 秒的确认避免了 20 分钟的跑题
2. **反向提问代替正向提问**——「这个方案在什么场景下出问题？」替代「这个方案好不好？」
3. **审查清单口头过一遍**——不依赖 AI review 的结果，自己用 checklist 确认
4. **状态快照的认真记录**——跨 session 协作时，状态快照是唯一可靠的上下文传递工具

## 你现在应该做什么？

1. 找一个你最近想做但一直在拖延的项目
2. 花 15 分钟做规划（MVP 范围、技术选型），按本文的 session 拆分方法拆成 3-5 个 session
3. 给第一个 session 命名，开始执行
4. 在每个 session 结束时，做两件事：存档关键输出到本地笔记 + 为状态快照准备材料
5. 项目完成后，回来重读这篇文章，对照你的项目反思：哪些心法用对了？哪些还没用上？

---

**要点总结**
- AI 协作出品的质量，**不取决于 AI 有多强，而取决于你的协作习惯有多好**
- 整个项目可以用「10 个独立 session + 状态快照串联」的方式来管理——AI 是执行者，你是架构师
- 每个阶段都建立**明确的 scope 边界**（今天的任务是什么、不做什么），AI 的输出更精准、更可控
- **反向提问**让 AI 帮你发现隐藏的问题；**够好原则**让你在正确的地方停下来
- 关键产出要**存档到本地笔记系统**，跨 session 的一致性问题要通过**共享文档**（如 config.py）解决
- 代码所有权的最终检查标准很简单：你能独立理解、独立修改、独立为这段代码负责
