# 04-测试：AI时代的质量保障

> 读完本文你将了解：为什么 AI 时代测试比以往更重要、怎么用 AI 帮你写测试、以及怎么建立"每步都验证"的开发习惯

---

## 你可能遇到过的问题

你用 AI 快速生成了很多代码，但发现了这些问题：

- **AI 写的代码经常有隐藏的 bug**——你测试了几个主要功能没问题，但边界条件一触发就报错
- **你改了 AI 生成的某段代码，结果其他地方也坏了**——但你完全没意识到，直到用户反馈
- **你觉得写测试很浪费时间**——代码都能跑了，为什么还要写测试？
- **你让 AI 帮你写测试**——但它写的测试也跑不过，你也不知道该信谁的

这些问题指向同一个结论：**AI 写得快，所以需要更快地验证**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| AI 生成的代码质量很高，不需要测试 | AI 生成的是**统计上最可能的代码**，不是**正确的代码**。两者有本质区别 |
| 写测试是浪费时间 | 测试不是花时间"检查"——是花时间**提前把 bug 找出来**。AI 写得快，bug 也出得快 |
| 测试是测试工程师的工作 | 在 AI 时代，**每个人都是自己代码的测试工程师**——因为 AI 不会替你做测试 |
| 运行一下程序不报错就是没问题 | 程序不报错 ≠ 程序正确。**可能只是你没触发到 bug 的路径** |

---

## 遇到这些问题怎么办？

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| AI 写的函数大部分时候对，但边界条件会出错 | 给 AI 一个 prompt："给这个函数写 5 个测试用例，覆盖正常输入和边界条件" |
| 改了代码之后其他功能坏了 | 每次 commit 前跑一遍**已有的测试**——任何测试失败都能马上发现 |
| 不知道测试应该写什么 | 让 AI 帮你分析代码，生成测试用例 |
| 测试也跑不过，不知道是代码的 bug 还是测试的 bug | 先确认测试本身是正确的（手动验证一个简单场景），再排查代码 |
| 不想写测试 | 从"让 AI 帮你写测试"开始——你只需要 review AI 写的测试是否合理 |

## 测试金字塔

最常用的测试分类方式：

```
      /\
     /  \         E2E 测试（端到端）— 少而贵
    /    \
   /──────\       集成测试 — 适中
  /        \
 /──────────\     单元测试（基础）— 多而快
```

### 单元测试

测试**一个函数、一个模块**在隔离环境下的行为。

```python
# 要测试的函数
def calculate_discount(price: float, is_vip: bool) -> float:
    if is_vip:
        return price * 0.8
    return price * 0.9

# AI 帮你写的单元测试
def test_calculate_discount():
    assert calculate_discount(100, True) == 80     # VIP 打 8 折
    assert calculate_discount(100, False) == 90    # 普通打 9 折
    assert calculate_discount(0, True) == 0        # 边界：金额为 0
```

### 集成测试

测试多个模块一起工作是否正常。

```python
def test_user_registration_flow():
    # 测试：注册 → 发验证邮件 → 验证 → 登录
    user_id = register_user("test@example.com")
    assert user_id is not None
    email = get_last_email()  # 应该收到了验证邮件
    assert "verify" in email.subject
    login_result = login("test@example.com", "password123")
    assert login_result.success
```

### E2E 测试

模拟用户操作，测试整个系统的行为。对初学者来说，单元测试足够用了。

## AI 时代测试的金三角

```
         AI 生成代码
           /    \
          /      \
         /        \
    AI 生成测试 ←── 人工验证
```

1. **AI 写代码** → 你 review（按之前文章的方法）
2. **AI 写测试** → 你 review 测试用例是否合理
3. **你运行测试** → 如果测试通过，confidence 提高
4. **发现问题** → 让 AI 修代码或修测试

---

## 怎么用 AI 写测试

### 基础用法：为已有代码生成测试

```text
请为以下函数生成 pytest 测试用例，覆盖正常输入、边界条件和异常情况：

def divide(a: float, b: float) -> float:
    if b == 0:
        raise ValueError("除数不能为 0")
    return a / b
```

AI 可能生成：

```python
def test_divide_normal():
    assert divide(10, 2) == 5.0

def test_divide_by_zero():
    with pytest.raises(ValueError):
        divide(10, 0)

def test_divide_negative():
    assert divide(-6, 3) == -2.0
```

### 进阶用法：测试驱动开发（TDD）风格

```text
我打算实现一个函数：输入一个 URL，返回这个 URL 是否可访问。
先帮我写测试用例，我再实现。
```

## 在 AI 工作流中融入测试

这是 Vibe Coding 的最佳实践——**每完成一步，就验证一步**：

```
1. 让 AI 生成代码
2. 让 AI 生成该代码的测试
3. 运行测试
4. 如果测试失败 → 回到步骤 1 修复
5. 测试通过 → git commit
6. 清空 AI 上下文
7. 开始下一个功能
```

**关键：** 不要跳过第 2 步。AI 生成的测试可能不完善，但**有测试比没有测试好一万倍**。

---

## 你现在应该做什么？

1. 找一个你用 AI 写的函数，让 AI 为它生成 pytest 测试
2. 运行测试——看看有没有你没想到的边界情况
3. 如果所有测试通过，confidence 升级
4. 养成习惯：每次让 AI 生成代码时，**顺带让它生成测试**

**要点总结**
- AI 生成的是"看起来像对的代码"，不是"真的对的代码"——**测试比以往更重要**
- 测试金字塔：**单元测试**（多而快）> 集成测试 > E2E 测试（少而贵）
- 入门从 **pytest** 开始——简单、强大、AI 友好
- Vibe Coding 的黄金法则：**每完成一步，跑一遍测试，再 commit**
- AI 帮你写测试，你 review 测试是否合理——这是人机协作的最佳结合点之一
