# 03-AI开发安全入门：你必须避开的常见坑

> 读完本文你将了解：AI 生成代码最容易引入哪些安全漏洞、AI 开发特有的安全风险、怎么用 AI 反过来做安全检查、以及 CS 学生必须养成的基础安全习惯
> 注：本文面向个人开发者与学生，不讲复杂攻防，只讲最容易踩、后果最严重的基础安全问题，避开 90% 的常见风险。

---

## 你可能遇到过的问题

你用 AI 快速写完了项目，功能跑通了，但心里总有点不踏实：

- **AI 写的代码会不会藏着安全漏洞？** 比如 SQL 注入、文件上传漏洞，自己又看不出来
- **不小心把 API Key 推到了 GitHub**，没过多久就收到了盗刷提醒，不知道问题出在哪
- **把项目代码、论文草稿直接粘给公共大模型优化**，担心数据会不会被模型拿去训练
- **做的 Agent 能执行本地命令**，不知道会不会被恶意提示词诱导删文件、读隐私数据
- **觉得安全是大公司的事**，自己的小项目没人看，不会有人攻击
- **不知道怎么判断 AI 给的代码安不安全**，只能凭感觉用

有安全意识是非常重要的职业素养。AI 时代写代码变快了，但安全坑也变多了，很多问题都是学生高频踩中的。

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| AI 很聪明，写的代码自动考虑安全 | AI 优先保证功能可用，默认不会主动做安全加固；训练数据里本身就有大量带漏洞的代码，它会直接模仿 |
| 我的小项目没人用，不会被攻击 | 公开仓库的 API Key、漏洞是机器人批量扫描的，几小时内就会被探测到，和项目大小无关 |
| 安全就是防黑客攻击 | 安全还包括数据隐私、配置管理、依赖合规、敏感信息防护，范围远不止攻防 |
| 提示词注入只是小问题，不用在意 | 在 Agent 工具调用场景下，提示词注入可能直接导致命令执行、数据泄露，是 AI 应用的核心风险 |
| 安全会拖慢开发速度 | 基础安全习惯养成了只是举手之劳；出了问题再补救，成本是几十上百倍 |

## 常见问题快速解决

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 担心 AI 写的代码有注入漏洞 | 明确告诉 AI：「使用参数化查询，禁止拼接 SQL/命令字符串」，写完再让 AI 做一次安全自查 |
| 不小心把密钥推到了 GitHub | 第一时间去对应平台吊销旧密钥、生成新密钥；仓库历史里的泄露记录也要清理 |
| 不确定项目依赖有没有漏洞 | Python 用 `pip audit`、Node.js 用 `npm audit` 一键扫描依赖的已知安全问题 |
| AI 推荐的第三方包不敢装 | 先去官方包仓库核对包名、下载量、更新时间，警惕拼写相似的恶意包 |
| 想快速检查代码安全性 | 把代码发给 AI，用标准安全审查 prompt 让它扫描常见漏洞 |
| 怕提示词注入搞坏 Agent | 给系统提示词加安全约束，工具调用只开放最小必要权限，敏感操作加人工确认 |

## AI 生成代码最常带出的 4 类通用漏洞

AI 的训练数据来自海量开源代码，其中包含了大量存在安全问题的写法。如果不明确要求，AI 会默认生成「能跑但不安全」的代码，以下是最高发的几类。

### 1. 凭证硬编码（学生最高发）
这是最容易犯、后果也最直接的错误。AI 写示例代码时经常直接把密钥写死在代码里：
```python
# ❌ 危险写法：AI 示例代码里非常常见
api_key = "sk-xxxxxxxxxxxxxxxxxxxx"
client = OpenAI(api_key=api_key)
```
如果把这段代码推到公开仓库，批量扫描的机器人几小时内就能爬取到密钥，进而盗刷额度、调用接口。

**正确做法**：永远用环境变量 + `.env` 文件管理凭证，配合 `python-dotenv` 读取：
```python
# ✅ 安全写法
import os
from dotenv import load_dotenv
load_dotenv()

client = OpenAI(
    api_key=os.getenv("SILICONFLOW_API_KEY"),
    base_url="https://api.siliconflow.cn/v1"  # 国内直接用，无需特殊网络
)
```
同时务必把 `.env` 加入 `.gitignore`，确保永远不会被提交到代码仓库。GitHub 现在自带**推送保护（Push Protection）**功能，检测到密钥会自动拦截提交，建议开启。

### 2. 注入类漏洞（SQL/命令注入）
AI 写查询逻辑时，图省事会直接用字符串拼接，这是最经典的安全漏洞：
```python
# ❌ 危险写法：存在 SQL 注入风险
query = f"SELECT * FROM users WHERE username = '{user_input}'"
cursor.execute(query)
```
恶意用户输入 `' OR 1=1 --` 就能绕过验证，甚至拖走整个数据库。

**正确做法**：要求 AI 使用参数化查询，由数据库驱动处理输入：
```python
# ✅ 安全写法
query = "SELECT * FROM users WHERE username = ?"
cursor.execute(query, (user_input,))
```
除了 SQL 注入，命令执行、模板渲染场景也会有同类注入问题，核心原则是：**永远不要把用户输入直接拼进可执行的代码/命令里**。

### 3. 输入校验完全缺失
AI 写接口、文件上传功能时，经常默认所有输入都是合法的，不做任何校验：
```python
# ❌ 危险写法：没有任何校验
@app.post("/upload")
async def upload(file: UploadFile):
    content = await file.read()
    with open(f"./uploads/{file.filename}", "wb") as f:
        f.write(content)
```
这段代码可以被上传任意恶意脚本，甚至通过 `../` 路径穿越写到系统任意位置。

**正确做法**：所有用户输入都不可信，必须校验类型、大小、格式、范围：
```python
# ✅ 安全写法：校验文件类型、大小、清理文件名
ALLOWED_EXTENSIONS = {"png", "jpg", "pdf"}
MAX_SIZE = 10 * 1024 * 1024  # 10MB

@app.post("/upload")
async def upload(file: UploadFile):
    # 校验文件大小、后缀、清理文件名
    if file.size > MAX_SIZE:
        return {"error": "文件过大"}
    ext = file.filename.split(".")[-1].lower()
    if ext not in ALLOWED_EXTENSIONS:
        return {"error": "不支持的文件类型"}
    safe_filename = secure_filename(file.filename)
    # ... 后续保存逻辑
```

### 4. 拼写劫持的恶意依赖
AI 推荐第三方库时，可能会给出名字和知名库非常像的恶意包（Typosquatting 攻击），比如把 `requests` 写成 `request`，把 `numpy` 写成 `numpi`。这类包已经在 PyPI、npm 等平台出现过很多次，安装后会窃取数据、执行恶意代码。

**防范方法**：
- 安装前先去官方包仓库核对准确包名
- 优先选用下载量高、维护活跃的官方包
- 不要安装来源不明、搜索不到官方仓库的包

## AI 开发特有的安全风险

除了通用代码漏洞，做 AI 应用还会遇到传统开发没有的新型安全问题，这部分最容易被忽略。

### 1. 提示词注入
用户通过输入恶意内容，绕过你的系统提示词，操控大模型执行预期外的操作。
- 普通场景：让 AI 忽略系统指令、输出违规内容
- 危险场景：在 RAG 应用里嵌入恶意提示，诱导 AI 泄露知识库敏感内容
- 高危场景：Agent 能调用工具时，被诱导执行删除文件、调用付费接口等危险操作

**基础防护**：
- 系统提示词里增加安全约束，明确禁止执行的操作
- Agent 工具遵循最小权限原则，高危操作（删文件、调用付费接口）加二次确认
- 对用户输入做基础的恶意内容检测

### 2. 敏感数据外泄
很多人图省事，会把未公开的项目代码、论文草稿、个人隐私数据、公司内部文档直接粘贴到公共大模型里优化。绝大多数公共大模型的服务条款里，会保留输入内容用于训练迭代，可能造成数据泄露。

**防范原则**：
- 敏感、未公开、涉密的数据，不要粘贴到公共大模型
- 处理隐私数据优先用本地部署的开源模型，或企业级合规的 API
- 调用第三方大模型 API 时，不要传入不必要的个人敏感信息

### 3. Agent 工具调用越权
当你的 AI Agent 具备执行本地命令、读写文件、调用系统接口的能力时，如果没有做权限限制，一旦被提示词注入突破，风险会非常大。

**防护原则**：
- 最小权限：Agent 只给完成任务必须的权限，比如只能读写指定目录、只能调用指定接口
- 操作白名单：只允许执行预设好的安全命令，禁止任意命令执行
- 高危操作确认：删除、修改、付费类操作，必须有二次确认机制

## 用 AI 做安全审查：让 AI 查它自己写的代码

AI 既是安全风险的来源，也是非常好用的安全检查工具。它能快速识别绝大多数常见漏洞模式，效率远高于人工逐行看。

### 标准工作流
```
1. AI 生成功能代码
2. 把代码发给 AI，执行安全审查
3. AI 列出发现的风险点、严重等级、修复方案
4. 你确认后，让 AI 修复对应问题
5. 最终再人工核对核心风险点
```

### 通用安全审查 Prompt 模板
```
请对以下代码做全面的安全审查，按严重程度从高到低列出所有风险，覆盖以下维度：
1. 注入类风险：SQL 注入、命令注入、XSS、提示词注入
2. 凭证安全：是否有硬编码的密钥、密码、Token
3. 输入校验：所有外部输入是否做了合法性校验
4. 权限控制：是否存在越权访问、路径穿越风险
5. 信息泄露：错误信息、日志是否会泄露内部敏感信息
6. 依赖安全：是否使用了有已知风险的依赖或不安全的写法

每个风险点请说明：风险位置、风险原理、修复建议。
```

### 专项场景检查
- **API 接口专项**：额外检查鉴权、速率限制、参数校验
- **Agent 专项**：额外检查工具调用权限、提示词注入防护、敏感操作兜底
- **数据处理专项**：额外检查隐私数据脱敏、数据存储合规

> 💡 提示：不要完全依赖 AI 的安全审查。它能发现 80% 的常见问题，但复杂的逻辑漏洞、业务层面的安全问题依然需要人工判断。

## CS 学生必须养成的安全习惯

不用一开始就学复杂的攻防，把基础习惯养好，就能避开 90% 的安全坑。

| 优先级 | 安全习惯 | 具体说明 |
|--------|---------|---------|
| **P0 红线（绝对不能碰）** | 凭证绝不进代码 | API Key、密码、Token 永远用环境变量管理，绝不硬编码写进代码 |
| **P0 红线** | 敏感文件不进 Git | `.env`、密钥文件、本地配置必须加入 `.gitignore`，提交前确认 |
| **P0 红线** | 敏感数据不喂公共大模型 | 未公开代码、论文、隐私数据，不要粘贴到公共 AI 工具里 |
| **P1 基础（写代码必做）** | 所有输入必须校验 | 任何来自用户/外部的内容都不可信，类型、大小、范围都要校验 |
| **P1 基础** | 遵循最小权限原则 | 代码、数据库、Agent 工具都只给完成任务必须的最小权限 |
| **P1 基础** | 依赖包名仔细核对 | 安装第三方库前确认包名正确，警惕拼写相似的恶意包 |
| **P2 进阶（养成更好）** | 定期扫描依赖漏洞 | 用 `pip audit` / `npm audit` 定期检查依赖的已知安全问题 |
| **P2 进阶** | AI 代码必做安全自查 | 每段 AI 生成的核心代码，提交前过一次安全审查 |
| **P2 进阶** | 错误信息不泄露细节 | 对外报错只给通用提示，不要暴露数据库结构、文件路径等内部信息 |

## 你现在应该做什么？

1. 检查自己所有公开的 GitHub 仓库，看看有没有泄露过 API Key、密码；仓库的「Security」标签页自带密钥扫描结果
2. 给最近一个 AI 写的项目补全 `.gitignore`，确认 `.env`、本地配置文件都在忽略列表里
3. 找一段 AI 生成的接口/功能代码，用文中的安全审查 prompt 跑一次，看看能发现多少问题
4. 想一下你正在做的 AI 项目，有没有提示词注入、数据泄露的风险，能做哪些基础防护

**要点总结**
- AI 优先保证功能可用，**不会主动考虑安全**，训练数据里的漏洞写法它会直接模仿，不能默认 AI 写的代码就是安全的
- 学生最高发的四类通用安全问题：**密钥硬编码泄露、注入攻击、缺失输入校验、恶意依赖**，都是可以通过基础习惯避免的
- AI 开发有特有风险：**提示词注入、敏感数据外泄、Agent 权限失控**，做 AI 应用时必须额外关注
- AI 既是风险来源，也是安全工具：用标准 prompt 可以让它帮你做代码安全审查、修复漏洞
- 安全不是大公司的事，是基础职业习惯。记住三个核心原则：**凭证不进代码、输入永远校验、最小权限原则**
