# 03-安全与AI：你需要注意什么

> 读完本文你将了解：AI 生成的代码可能引入哪些安全漏洞、怎么用 AI 帮你做安全检查、以及作为 CS 学生应该掌握的基本安全观念

---

## 你可能遇到过的问题

你用 AI 生成了代码，项目跑起来了，功能也有——但你开始担心：

- **AI 写的代码会不会有安全漏洞？**——万一有人利用这个漏洞搞破坏怎么办？
- **你看到新闻说 AI 生成的代码有安全问题**——但不知道具体是什么问题
- **你把 API Key 放在代码里上传到了 GitHub**——然后收到了一封账单通知
- **你不知道怎么判断 AI 给你的代码是否安全**

这些问题说明你已经开始有安全意识了——这是 CS 学生最重要的素质之一。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| AI 很聪明，不会写出有安全问题的代码 | AI 的训练数据中充满了 GitHub 上有安全问题的代码。AI 会模仿这些模式，包括漏洞 |
| 安全是大公司才需要考虑的事 | **个人项目也经常被扫描**——GitHub 上公开的 API Key 几小时内就可能被机器人爬走 |
| 我的项目没人用，不会有安全问题 | 安全不是"有没有人用"的问题——是**习惯问题**。现在不养成好习惯，以后做真实项目也会犯同样的错 |
| 安全就是不要被黑客攻击 | 安全还包括：数据隐私、配置管理、依赖管理、日志不泄露敏感信息 |

---

## 遇到这些问题怎么办？

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 担心 AI 写的代码有 SQL 注入 | 告诉 AI "用参数化查询，不要拼接 SQL 字符串"——然后检查 AI 生成的代码是否真的没有拼接 |
| 把 API Key 传到了 GitHub | 立即去平台吊销 Key，生成新的。然后把 `.env` 加到 `.gitignore`（参考环境准备篇 03） |
| 不知道项目依赖有没有漏洞 | 用 `pip audit` 或 `npm audit` 检查依赖的安全问题 |
| AI 让你安装不明包 | 检查包名是否真实（想想有没有 typosquatting 攻击） |
| 不确定代码的安全性 | 把代码发给 AI 做安全审查："请找出这段代码中的所有安全漏洞" |

## AI 时代最常见的几个安全问题

### 1. API Key 和凭证泄露（最常见）

这是 CS 学生最容易犯的安全错误。AI 可能生成这样的代码：

```python
# AI 生成的代码——Key 直接硬编码在代码里
api_key = "sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

如果你把这个文件直接推到 GitHub，你的 Key 就暴露了。

**正确做法：** 始终用环境变量或 `.env` 文件管理凭证（详见环境准备篇 01 和 03）。

### 2. 注入攻击

AI 经常生成这样的代码：

```python
# 不安全的写法——AI 经常这么写
query = f"SELECT * FROM users WHERE name = '{user_input}'"
```

这段代码容易受到 SQL 注入攻击——用户输入 `' OR 1=1 --` 就能绕过验证。

**正确做法：** 要求 AI 用参数化查询：

```python
# 安全的写法——告诉 AI 用这个
query = "SELECT * FROM users WHERE name = ?"
cursor.execute(query, (user_input,))
```

### 3. 输入验证缺失

AI 生成的 API 端点经常不验证输入数据：

```python
@app.post("/upload")
async def upload_file(file: UploadFile):
    # AI 没有检查文件大小、类型、文件名安全性
    content = await file.read()
    with open(f"/tmp/uploads/{file.filename}", "wb") as f:
        f.write(content)
```

这段代码允许用户上传任意文件——包括恶意脚本。

**正确做法：** 始终验证输入。告诉 AI "添加输入验证：检查文件类型、限制文件大小、清理文件名"。

### 4. 不安全的依赖

AI 可能会推荐你安装一个看起来像知名包的库——但名字稍微不同（如 `python-solana` vs `solana-py` 的 typosquatting）。这种情况确实发生过。

**正确做法：** 安装前检查包的下载量、维护时间、GitHub 仓库。用 `pip install` 前先确认包名正确。

## 用 AI 做安全审查

AI 在安全检查方面其实很有用——它能发现很多常见的漏洞模式。

### 一个完整的工作流

```text
你：这是一段用户登录的代码，请做安全审查

AI：发现了几个问题：
1. SQL 注入风险——第 12 行直接拼接用户输入
2. 密码以明文存储——应该用哈希
3. 没有登录频率限制——可能被暴力破解
4. 错误信息泄露了数据库结构——第 15 行的错误信息太详细

你：好的，帮我把这些问题修复
```

### 安全审查 Prompt 模板

```
请对以下代码做安全审查，关注：
- 注入攻击（SQL、命令、XSS）
- 凭证管理（API Key 是否硬编码）
- 输入验证（是否检查了用户输入）
- 输出编码（是否防止 XSS）
- 依赖安全（是否有已知漏洞的依赖）
- 敏感信息泄露（错误信息、日志）
```

## CS 学生应该养成的安全习惯

| 优先级 | 习惯 | 说明 |
|--------|------|------|
| P0 | 凭证不进代码 | API Key、密码、Token **永远**不写死在代码里 |
| P0 | .gitignore 配好 | 确保 `.env`、`secrets/`、`config.json` 不会被提交 |
| P1 | 输入验证 | 用户输入不可信——任何来自用户的数据都要验证 |
| P1 | 最小权限 | 代码只需要什么权限就给什么权限——不要用 root |
| P2 | 依赖审查 | 定期检查依赖是否有已知漏洞 |
| P2 | 安全 Code Review | 每次 AI 生成代码后，让它做一次安全审查 |

---

## 你现在应该做什么？

1. 检查你所有 GitHub 仓库——有没有暴露过 API Key？（有一个叫 `git secrets` 的工具可以帮你扫）
2. 检查 `.gitignore`——确保 `.env` 在忽略列表中
3. 找一个 AI 帮你生成过的代码，让它做一次安全审查
4. 养成习惯：每次 AI 写完代码，加一句"请检查这段代码的安全性"

**要点总结**
- AI 的训练数据中包含了很多有安全漏洞的代码——**AI 会模仿这些漏洞**
- 最常见的 AI 引入的安全问题：**凭证泄露、注入攻击、输入验证缺失、不安全的依赖**
- 用 AI 做安全审查：让 AI 检查它自己生成的代码
- **三个最重要的习惯：凭证不进代码、.gitignore 配好、输入永远要验证**
