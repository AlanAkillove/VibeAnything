# 02-系统设计思维入门

> 读完本文你将了解：什么是系统设计思维、即使做小项目也需要它、怎么用模块化的方式组织代码、以及 AI 怎么帮你做设计决策

---

## 你可能遇到过的问题

你用 AI 写了一些项目——从简单的脚本到稍微复杂的应用。但你发现：

- **你的项目代码全都堆在同一个文件里**——AI 每次都把新功能加到最后，文件越来越长，越来越看不懂
- **改一个功能，另外一个不相关的地方就报错了**——代码之间像蜘蛛网一样纠缠在一起
- **你想复用之前写的某个功能**，但发现它和别的代码紧紧耦合在一起，根本拆不出来
- **你不知道什么样的代码组织方式是"好的"**——AI 写什么你就用什么

这些问题不是"代码写得不对"，而是**缺乏系统设计的思维**。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 系统设计是大公司才需要的东西 | **任何超过 200 行的项目都需要设计思维**——不一定是微服务架构，但至少要做好模块划分 |
| 有 AI 写代码，不需要设计 | AI 最擅长在**已有的框架里填空**。如果框架本身是乱的，AI 只会让乱得更快 |
| 设计就是把代码拆成多个文件 | 拆文件只是手段。真正的设计是**关注点分离**：每个模块只做一件事，做好 |
| 系统设计很抽象，等以后再用 | **第一门编程课学到的函数、类就是设计的起点**——把功能封装成独立的单元 |

---

## 遇到这些问题怎么办？

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 代码全在一个文件里 | 把不同功能拆到不同函数/文件。告诉 AI："把这个功能独立成一个模块" |
| 改一个地方别的地方就坏 | 模块之间通过**接口**通信，而不是共享全局变量。让 AI 帮你提取接口 |
| 想复用代码但拆不出来 | 这是耦合的表现。让 AI 分析依赖关系，逐步解耦 |
| 不知道怎么组织项目 | 从常见模式开始：按功能分目录、用 `main.py` 做入口、配置单独放 |

---

## 什么是系统设计思维？

系统设计思维是**在写代码之前，先想清楚代码怎么组织**的能力。

它关注的是：

| 问题 | 设计的角度 |
|------|-----------|
| 这个功能应该放在哪个文件？ | 模块划分 |
| 模块 A 和模块 B 怎么通信？ | 接口设计 |
| 如果以后要加新功能，怎么做？ | 可扩展性 |
| 如果数据量变大，代码还能跑吗？ | 可伸缩性 |
| 别人能看懂我的代码吗？ | 可读性 |

即使你的项目只有几百行代码，这些问题的答案也会影响你能不能高效地往下加功能。

---

## 三个核心概念

### 1. 关注点分离（Separation of Concerns）

> 每个模块只关心自己负责的那一部分。

一个简单的例子——AI 聊天应用：

```
不好的设计：
app.py  →  包含 UI 逻辑 + API 调用 + 消息处理 + 文件保存

更好的设计：
ui.py        → 只处理用户界面
api.py       → 只处理 AI API 调用
messages.py  → 只处理消息格式和存储
config.py    → 只存配置信息
```

每个文件只负责一件事。想改 UI 样式，不需要碰 API 调用的代码。

**AI 可以帮助你**：给它你的代码，让它帮你拆分成模块：

> "帮我把这个脚本按功能拆分成模块，每个模块一个文件，并告诉我每个文件的职责"

### 2. 接口（Interface）

> 接口是模块之间的"合同"——规定好输入和输出，内部怎么实现在合同后面随便改。

一个简单的接口定义：

```python
def search_users(keyword: str) -> list[dict]:
    """
    根据关键词搜索用户。
    参数: keyword - 用户名关键词
    返回: 匹配的用户列表，每个用户包含 id、name、email
    """
    pass  # 实现细节暂时不管
```

接口写好了，其他人（或者 AI）就可以基于这个接口独立工作——不需要知道内部实现。

### 3. 分层（Layers）

> 把系统从上到下分成几层，每层只和相邻的层通信。

最常见的分层：

```
用户界面层（UI）
    ↓
业务逻辑层（Logic）
    ↓
数据访问层（Data）
```

UI 层不直接操作数据库，数据层不关心按钮怎么显示。每一层只做自己的事。

## 从做小项目开始练习

系统设计不是只有"设计百万用户系统"才叫设计。从第一个项目就可以开始练习：

### 项目：一个简单的笔记应用

**第 1 版（没有设计思维）：**

```python
# notes.py — 300 行，UI、存储、搜索全在一起
def add_note():
    # ... 50 行 UI 代码
    # ... 30 行文件操作
    # ... 20 行格式处理

def search_notes():
    # ... 又有一堆 UI 和逻辑混在一起
```

**第 2 版（初步设计）：**

```python
# main.py — 入口，只调用其他模块
from ui import run_app
run_app()

# ui.py — 只处理用户输入和显示
# storage.py — 只处理文件的读写
# search.py — 只处理搜索逻辑
```

这时你想加新功能（比如标签分类），你只需要在 `storage.py` 里加一个字段，在 `search.py` 里加一个过滤条件——不需要改 UI 代码。

## AI 时代的系统设计

### AI 帮不了你什么

AI 不会替你做出这些设计决策：

- "这个功能应该放在哪个模块？"
- "这个接口应该暴露哪些方法？"
- "现在拆分成这样，以后扩展会不会不方便？"

这些都是**人的判断**。

### AI 能帮你什么

| 场景 | 可以问 AI |
|------|-----------|
| 代码重构 | "帮我把这段 500 行的脚本拆分成模块" |
| 接口设计 | "这个功能应该提供什么样的接口？" |
| 设计评审 | "看看我的项目结构，有什么改进建议？" |
| 生成骨架 | "给我一个 Flask 项目的基础目录结构" |
| 依赖分析 | "分析一下这个项目中哪些模块之间有循环依赖" |

**关键：** AI 能在你做出设计决策后，帮你快速生成符合这个决策的代码框架。但做决策的人是你。

---

## 你现在应该做什么？

1. 找一个你最近用 AI 写的项目（哪怕只有几百行）
2. 让 AI 分析现在的代码组织方式："分析这个项目的模块划分，有什么改进空间？"
3. 根据 AI 的建议，把代码拆分成模块
4. 定义一个接口，让模块之间通过接口通信
5. 下次加新功能时，先想清楚它属于哪个模块，再让 AI 实现

**要点总结**
- **系统设计思维**不是大公司的专利——200 行的项目也需要
- 三个核心概念：**关注点分离、接口、分层**
- AI 能帮你**实现**设计决策，但不能帮你**做出**设计决策
- 好的设计让加新功能变得容易，坏的设计让修 bug 都困难
- 从小项目开始练习设计思维——不需要等到做百万用户系统才学
