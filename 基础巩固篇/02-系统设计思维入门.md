# 02-系统设计思维入门：AI时代的代码组织能力

> 读完本文你将了解：什么是系统设计思维、为什么小项目也需要它、三个核心设计原则、怎么用模块化约束AI生成的代码、以及AI在设计中能帮你做什么、不能替你做什么
> 注：本文不讲分布式、微服务等大型架构，只讲每个开发者都该掌握的基础设计思维，几百行的小项目也能用得上。

---

## 你可能遇到过的问题

你用 AI 快速写了不少项目，从简单脚本到完整的 RAG 应用、Agent 工具，但很快会遇到这些瓶颈：

- **AI 写代码只会往后堆**——所有功能都塞在同一个文件里，几百行后逻辑缠成一团，自己都看不懂
- **改一个小功能，不相关的地方跟着报错**——代码之间像蜘蛛网一样耦合，动一处就牵一发而动全身
- **想复用之前写的大模型调用逻辑**——发现它和 UI、存储、业务逻辑缠在一起，根本拆不出来
- **不知道什么样的代码是「好的」**——AI 怎么写你就怎么用，加着加着就变成了维护不动的屎山
- **觉得系统设计是大公司的事**——自己写小项目，能跑就行，不需要搞什么设计

这些问题本质都不是「代码写得不对」，而是**缺乏最基础的系统设计思维**。AI 写代码越快，设计能力的重要性就越突出——没有好的框架，AI 只会让代码乱得更快。

---

## 常见的误解

| 你以为的 | 实际的 |
|---------|--------|
| 系统设计是大厂百万用户系统才需要的东西 | **任何超过 200 行的项目都需要设计思维**——不用微服务，但至少要做好模块划分、职责分离 |
| 有 AI 写代码、能重构，不用提前设计 | AI 最擅长在已有框架里填内容。如果框架本身是乱的，AI 重构只会越改越乱，还会引入隐藏 bug |
| 设计就是把代码拆成多个文件 | 拆文件只是表面。真正的设计是**关注点分离**：每个模块只做一件事，通过明确的接口通信 |
| 设计就是过度工程化，先写了再说 | 前期花 10 分钟做简单设计，后期改功能能省几个小时。越先写，后面返工成本越高 |
| 系统设计很抽象，以后工作了再学 | 函数、类、模块划分，就是设计的起点。从第一个小项目开始练，能力是慢慢长出来的 |

---

## 常见问题快速解决

| 你遇到的问题 | 现在可以怎么做 |
|-------------|---------------|
| 所有代码堆在一个文件里 | 告诉 AI：「按功能把这段代码拆成独立模块，每个模块一个文件，明确每个文件的职责」 |
| 改一处就崩其他地方 | 让 AI 帮你提取模块接口，模块之间只通过接口通信，禁止互相修改内部变量 |
| 想复用的代码拆不出来 | 让 AI 分析代码的依赖关系，把通用逻辑抽成独立的工具函数/类，解除业务耦合 |
| 不知道项目该怎么组织目录 | 从通用模板起步：入口文件、配置、业务逻辑、数据层、工具函数分目录存放 |
| 不确定自己的设计好不好 | 把项目结构发给 AI，让它从「职责单一、耦合度、扩展性」三个维度做评审 |

## 什么是系统设计思维？

系统设计思维，本质是**写代码之前，先想清楚代码怎么组织**的能力。它不关心具体某一行代码怎么写，关心的是整体结构：

| 核心问题 | 设计维度 |
|---------|---------|
| 这个功能该放在哪个模块里？ | 模块划分与职责边界 |
| 两个模块之间该怎么传递数据？ | 接口设计 |
| 以后加新功能，会不会要推翻重写？ | 可扩展性 |
| 数据量涨 10 倍，架构还撑得住吗？ | 可伸缩性 |
| 别人接手能不能快速看懂？ | 可读性与可维护性 |

哪怕你的项目只有几百行，这些问题的答案，也直接决定了你后续加功能、改 bug 的效率。AI 时代尤其如此：AI 可以帮你快速写实现，但如果没有清晰的设计约束，它生成的代码只会不断堆砌，很快就会失控。

## 三个核心设计原则

这三个原则是所有系统设计的基础，小到单文件脚本，大到分布式系统，都适用。

### 1. 关注点分离：每个模块只做一件事

> 核心思想：把不同职责的代码拆开，每个模块只负责自己的那一部分，互不干扰。

最典型的反面例子，就是 AI 写的「全能脚本」：一个文件里同时有用户交互、大模型调用、文件读写、数据处理，改 UI 都可能不小心改坏 API 调用逻辑。

以一个 AI 聊天工具为例：
```
❌ 糟糕的设计（全堆在一起）
app.py → 包含命令行UI + 大模型API调用 + 消息历史存储 + 提示词模板

✅ 更好的设计（按职责拆分）
main.py          → 程序入口，只负责启动
ui/cli.py        → 只处理用户输入输出，不碰业务逻辑
llm/client.py    → 只封装大模型 API 调用，统一管理密钥、重试、错误
memory/store.py  → 只负责消息历史的读写存储
prompts/base.py  → 只存放提示词模板，方便统一修改
```

拆分之后，想换大模型厂商，只需要改 `llm/client.py`，不会影响 UI 和存储；想改提示词，只需要动模板文件，不用翻业务代码。

**AI 辅助用法**：把你的单文件脚本发给 AI，说「按照关注点分离原则，把这段代码拆成多个模块，每个模块一个文件，说明每个文件的职责」。AI 可以帮你快速完成拆分，但拆得合不合理，需要你自己判断。

### 2. 接口契约：模块之间的「约定」

> 核心思想：模块之间通过明确的接口通信，约定好输入输出格式，内部实现随便改，不影响外部。

接口就像两个模块之间的合同：只要输入输出不变，内部怎么优化、怎么重构都可以，不会牵连其他模块。这对于 AI 协作尤其重要——你定好接口，AI 只在接口内部写实现，就不会乱改边界、把代码写得一团糟。

一个简单的接口示例：
```python
# llm/client.py 对外暴露的接口
def chat(messages: list[dict], model: str = "deepseek-v4-flash") -> str:
    """
    调用大模型对话接口
    :param messages: 对话历史，格式为 [{"role": "user", "content": "..."}]
    :param model: 模型名称
    :return: 模型回复文本
    """
    # 内部的鉴权、重试、错误处理，外部都不需要关心
```

只要这个接口的参数和返回值不变，你内部把 DeepSeek 换成 Qwen、加缓存、加重试逻辑，所有调用它的地方都不用改。

### 3. 分层架构：从上到下，逐层调用

> 核心思想：把系统按职责分成上下几层，上层只能调用下层，不能跨层、不能反向调用。

最经典的三层架构，套用到 AI 应用上就是：
```
┌─────────────────────┐
│  交互层（UI）        │  命令行 / 网页界面，只负责和用户交互
└─────────┬───────────┘
          ↓
┌─────────────────────┐
│  业务逻辑层          │  Agent 编排、RAG 流程、业务规则，核心功能逻辑
└─────────┬───────────┘
          ↓
┌─────────────────────┐
│  能力层              │  大模型调用、向量检索、工具函数，封装基础能力
└─────────┬───────────┘
          ↓
┌─────────────────────┐
│  数据层              │  文件读写、数据库操作，只管数据存取
└─────────────────────┘
```

分层的好处是逻辑清晰：UI 层不直接操作数据库，数据层不关心业务规则。以后想把命令行换成网页，只需要改最上层的交互层，下面所有逻辑都可以复用。

## 从小项目开始练习：RAG 问答机器人

系统设计不是只有做百万用户系统才叫设计，从第一个小项目就可以练手。我们以一个最简单的 RAG 问答机器人为例：

### 第 1 版：无设计，AI 随手写
```
rag.py （400 行一个文件）
- 读取 PDF 文件
- 切分文本
- 调向量化接口
- 存到本地向量库
- 接收用户问题
- 检索相关片段
- 拼提示词调大模型
- 打印结果
```
所有逻辑堆在一起，想换成 Word 文档、想换向量库、想换大模型，都要在几百行里翻找修改，很容易改出 bug。

### 第 2 版：基础设计，按职责拆分
```
project/
├── main.py              # 程序入口
├── config.py            # 配置、API Key、参数
├── document/            # 文档处理层
│   ├── loader.py        # 读取各类文档
│   └── splitter.py      # 文本切分
├── vector/              # 向量检索层
│   ├── store.py         # 向量存储
│   └── retriever.py     # 检索逻辑
├── llm/                 # 大模型层
│   └── client.py        # 模型调用封装
└── pipeline/            # 业务层
    └── rag_pipeline.py  # 完整 RAG 流程编排
```

拆分之后，想换文档格式，只改 `document/loader.py`；想换向量数据库，只改 `vector/store.py`；业务流程调整，只动 `pipeline` 里的逻辑。每个模块职责单一，AI 写代码也有明确的边界，不会乱写。

## AI 时代的设计：AI 是执行者，不是决策者

Google 在官方代码评审指南里明确提出，代码评审首要考察的就是**设计**：代码是否整体设计合理、是否适合当前系统、复杂度是否可控。AI 可以帮你完成大量实现工作，但设计决策永远要由人来做。

### AI 替你做不了的核心决策
- 模块边界划在哪里：哪些逻辑该放一起，哪些该拆开
- 接口怎么定义：暴露哪些方法、输入输出是什么格式
- 技术选型权衡：用简单方案还是复杂方案，性能和维护成本怎么选
- 扩展性预判：现在的设计以后加新功能会不会费劲

这些都是需要人结合业务场景、数据规模、团队情况做判断的，AI 没有你的上下文，给不出最优解。

### AI 能帮你做的事
| 场景 | 具体用法 |
|------|---------|
| 生成项目骨架 | 「给我一个 Python RAG 项目的标准目录结构，说明每个目录的职责」 |
| 执行代码重构 | 「把这个单文件脚本按关注点分离原则拆成多个模块，保持功能不变」 |
| 辅助设计评审 | 「评审一下这个项目的模块设计，从职责单一、耦合度、扩展性三个维度说问题和改进建议」 |
| 检查复杂度 | 「分析这段代码有没有不必要的复杂度，能不能简化」 |
| 识别坏味道 | 「看看这段代码有没有循环依赖、全局变量滥用的问题」 |

一句话总结：**你定方向和框架，AI 填细节和执行**。不要反过来让 AI 定设计，你去填坑——那样只会越写越乱。

---

## 你现在应该做什么？

1. 找一个最近用 AI 写的项目（哪怕只有几百行）
2. 把项目代码发给 AI，让它分析当前的模块划分问题，给出改进建议
3. 根据建议，把代码按职责拆分成 2-3 个模块，定义好模块之间的接口
4. 试着加一个小功能，先想清楚该放在哪个模块，再让 AI 写实现代码
5. 对比拆分前后，加新功能、改 bug 的效率有没有提升

**要点总结**
- 系统设计不是大项目的专利，**200 行的小项目也需要基础设计思维**，AI 写代码越快，设计越重要
- 三个核心原则：**关注点分离、接口契约、分层架构**，是所有设计的基础
- AI 是设计的**执行工具**，能帮你拆分、重构、评审，但做不了核心决策，定框架的永远是人
- 好的设计让加新功能变轻松，坏的设计让修 bug 都困难；前期花 10 分钟设计，后期能省几倍的时间
- 不用等做大型系统才学，**从小项目开始练手**，每写一个项目都刻意优化结构，能力会自然提升
