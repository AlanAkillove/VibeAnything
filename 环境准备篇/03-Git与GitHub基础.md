# 03-Git 与 GitHub 基础

> 读完本文你将了解：为什么 AI 开发时代版本控制反而更重要、日常开发必用的 Git 核心命令、怎么把代码推到 GitHub、以及如何用 Git 管理 AI 生成的代码，避免改坏回不去
> 注：本文面向零基础新手，只讲日常 80% 场景会用到的核心操作，复杂命令按需再学即可。

## 你可能遇到过的问题

你用 AI 快速生成了很多代码，改了一版又一版，却很快遇到这些麻烦：

- **AI 一轮对话把整个文件改坏了**——想回到上一个能用的版本，却发现已经覆盖保存了
- **改了一堆东西跑不起来**，完全记不清一小时前都动了哪些地方
- **实验室和宿舍两台电脑写代码**，只能靠 U 盘/微信传来传去，版本混乱
- **想把自己的项目传到 GitHub 分享**，却不知道仓库、提交、推送都是什么意思
- **对 Git 本能抵触**——光看到 checkout、rebase、merge 一堆术语就头大，觉得自己学不会

其实日常开发 80% 的场景，只用 5 个核心命令就能完全覆盖。AI 写代码越快，你越需要 Git 当你的「后悔药」。

| 你以为的 | 实际的 |
|---------|--------|
| 自己一个人写代码不需要 Git | **AI 时代单人开发反而更需要 Git**——AI 改代码快，改坏也快，没有版本记录根本没法回溯 |
| Git 特别难，要背很多命令 | 日常开发只用 5 个高频命令就能覆盖绝大多数场景，复杂命令遇到了再查就行 |

| 有 AI 帮我写代码，不用管提交历史 | AI 没有版本意识，每轮对话都是全新上下文。**你的 commit 历史就是唯一的「撤销按钮」** |
| GitHub 都是公开代码，不能存私有项目 | GitHub 可以创建私有仓库，免费用户也能使用，还自带密钥扫描、安全检查功能 |

## 常见问题快速解决

| 你遇到的问题 | 快速操作 |
|-------------|---------|
| AI 改坏了某个文件，想恢复原样 | `git restore 文件名.py` 回到上一次提交的版本 |
| 改了一堆东西全不想要了，一键回退 | `git reset --hard HEAD` 回到最近一次提交的干净状态 |
| 想看看当前都改了哪些地方 | `git diff` 查看所有未提交的改动详情 |
| 代码写了一半想先存个档 | `git add .` + `git commit -m "草稿: 完成xx功能一半"` 先提交保存 |
| 本地代码推送到 GitHub 失败 | 先检查是否关联了远程仓库，再确认 GitHub 账号权限是否正常 |
| 远端有更新，本地要同步 | `git pull` 拉取最新代码到本地 |

## 什么是 Git？为什么 AI 时代它更重要？

Git 是一个**分布式版本控制系统**，本质就是给你的项目文件不断拍「快照」，每一次提交都永久保存一个版本，任何时候都能回溯到任意历史节点。

在 AI 辅助开发的场景下，Git 的价值被进一步放大：
1. **AI 生成速度快，出错也快**：30 秒生成 200 行代码，也可能下一轮就把原本能用的逻辑改崩，没有版本记录就彻底找不回来
2. **AI 没有全局记忆**：每开一轮新对话，AI 就忘了之前的约定，很容易改乱项目结构，Git 能帮你守住每一个「可用状态」
3. **安全兜底**：GitHub 自带推送保护（Push Protection），能自动检测你不小心提交的 API 密钥、敏感信息，拦截后提醒你删除，过去一年已拦截超 830 万次密钥泄露
4. **Vibe Coding 的基础**：AI 时代最舒服的开发节奏，就是每完成一个可验证的小步骤就提交一次，再开新的 AI 对话推进下一个功能，步步有兜底

截至 2026 年 4 月，Git 最新稳定版为 **2.54.0**，是全球开发者的通用标准工具。

## 第一步：安装与初始配置

### 1. 安装 Git
去官网 [git-scm.com](https://git-scm.com) 下载对应系统的安装包：
- **Windows**：安装时默认选项即可，确保勾选「Add Git to PATH」，安装完成后重启终端生效
- **macOS**：终端输入 `git --version`，按提示安装 Xcode 命令行工具即可
- **Linux**：用系统包管理器安装，如 `sudo apt install git`

安装完成后，终端输入 `git --version`，能输出版本号即安装成功。

### 2. 首次全局配置（必须做，否则无法提交）
第一次使用 Git 需要设置你的身份信息，提交时会记录到版本历史里：
```bash
# 设置用户名（可以用 GitHub 昵称）
git config --global user.name "你的昵称"

# 设置邮箱（建议和 GitHub 注册邮箱一致）
git config --global user.email "你的邮箱地址"
```

只需要配置一次，以后所有项目都会沿用。

## 核心命令：五个命令覆盖 80% 日常场景

你不需要背几十条命令，先掌握以下 5 个高频命令，足够应对单人开发的绝大多数情况。

### 1. 初始化仓库
```bash
git init
```
在项目文件夹里执行一次，Git 就会开始跟踪这个目录下的所有文件变更，生成隐藏的 `.git` 文件夹。

### 2. 查看当前状态
```bash
git status
```
**这是使用率最高的命令**。它会告诉你：当前在哪个分支、哪些文件被修改了、哪些已暂存、哪些还没提交。拿不准的时候就敲一下，永远不会错。

### 3. 暂存改动
```bash
git add 文件名   # 只暂存指定文件
git add .        # 暂存当前目录所有改动
```
`add` 就是把改动放进「暂存区」，相当于把要提交的东西先打包整理好，确认无误后再正式提交。

> 💡 好习惯：提交前先用 `git diff` 看一眼具体改了什么，尤其是 AI 生成的代码，避免 AI 偷偷改了其他文件、混入敏感信息。

### 4. 提交版本快照
```bash
git commit -m "feat: 实现用户注册接口"
```
`commit` 会把暂存区的所有改动永久保存为一个历史版本，生成一个唯一的版本号。
- 双引号里是提交说明，写清楚「做了什么」，方便以后回溯
- 建议养成简单的命名习惯：`feat:` 加新功能、`fix:` 修 bug、`docs:` 更新文档、`chore:` 工程配置改动

### 5. 推送到云端（GitHub）
```bash
git push
```
把本地的所有提交记录同步到 GitHub 远程仓库，既做云端备份，也方便跨设备同步、分享给他人。

## 补充：两个救命的回退命令

这两个是 AI 开发场景下最常用的「后悔药」，一定要记住：

```bash
# 恢复单个文件到上一个提交的状态
git restore 文件名.py

# 放弃所有未提交的改动，一键回到最近一次提交的干净状态
git reset --hard HEAD
```

> ⚠️ 注意：`git reset --hard` 会彻底删除未提交的改动，执行前确认清楚，最好先 `git status` 看一眼都有哪些改动。

## 一个完整的 AI 开发日常流程

配合 AI 写代码时，推荐遵循「小步提交」原则，每完成一个可验证的小节点就存一次档：

```bash
# 1. 开始写代码前，先看一眼当前状态
git status

# 2. AI 生成/修改代码后，检查改动
git diff

# 3. 测试功能正常，确认改动没问题，暂存所有文件
git add .

# 4. 提交版本，写清本次做了什么
git commit -m "feat: 完成大模型 API 调用封装"

# 5. 同步到 GitHub 云端备份
git push

# 6. 开新一轮 AI 对话，继续下一个功能
```

如果下一轮 AI 把代码改坏了，直接一键回退到上一个可用版本：
```bash
git reset --hard HEAD
```

## GitHub：把代码托管到云端

GitHub 是全球最大的代码托管平台，也是开发者的「线上作品集」。现在的 GitHub 已经深度集成 Copilot AI 能力，能帮你自动审查代码、修复漏洞、生成提交说明。

### 把本地项目推到 GitHub
1. 打开 [GitHub.com](https://github.com) 注册账号，右上角点击「New repository」创建新仓库
2. 填写仓库名，选择 Public / Private，**新手建议勾选「Add .gitignore」并选择 Python 模板**，自动生成忽略文件
3. 创建完成后，在本地项目终端执行以下命令关联远程仓库：
```bash
# 关联远程仓库地址（替换成你自己的仓库链接）
git remote add origin https://github.com/你的用户名/仓库名.git

# 第一次推送，绑定本地和远端分支
git push -u origin main
```
后续再提交，直接 `git push` 就行。

### 下载别人的开源项目
```bash
git clone https://github.com/用户名/仓库名.git
```
这会把整个项目的代码、完整提交历史都下载到本地，直接就能运行和修改。

## 必配：.gitignore 文件

`.gitignore` 的作用是告诉 Git：**这些文件不要跟踪，也不要提交**。AI 开发项目里有很多东西绝对不能传到 GitHub，必须配置好。

Python 项目标准 `.gitignore` 至少包含以下内容：
```gitignore
# 虚拟环境（每个人的环境都不一样，不能同步）
.venv/
venv/

# Python 缓存文件
__pycache__/
*.pyc
*.pyo

# 敏感信息（绝对不能上传！）
.env
*.key
*.pem

# IDE 配置文件
.vscode/
.idea/

# 系统临时文件
.DS_Store
Thumbs.db

# 日志文件
*.log
```

> 🚨 特别提醒：`.env` 里存着你的 API Key，一旦提交到公开仓库就会被爬虫扫描盗刷。哪怕是私有仓库也建议不要提交密钥。GitHub 自带推送保护功能，检测到密钥会自动拦截推送，但最好从源头就做好隔离。

## AI 时代的 Git 最佳实践

### Vibe Coding 标准工作流
这是目前 AI 辅助开发最成熟的协作模式：
1. 开一轮新的 AI 对话，明确告知当前要实现的单一功能
2. AI 生成代码后，本地运行验证功能是否正常
3. 用 `git diff` 检查所有改动，确认没有多余修改、没有敏感信息
4. 验证通过 → `git add` + `git commit` 保存版本
5. 清空 AI 对话上下文，开启下一个功能的开发

这样每一步都有清晰的可用节点，任何时候出问题都能快速回退，不会陷入「越改越乱」的恶性循环。

### 其他实用建议
- **提交粒度要小**：不要攒一大堆改动才提交，一个功能点一个 commit，回退的时候更灵活
- **提交说明要清晰**：别只写「更新」「修改」，写清楚改了什么，以后看历史的时候一眼就能找到对应版本
- **善用 AI 查问题**：遇到 Git 报错不用慌，直接把报错信息复制给 AI，它会告诉你怎么解决
